# 00 · 约定与状态机（契约脊柱）

> **本文是所有域契约的共享地基。** 路由/版本、响应包络、错误信封、幂等、SSE 帧、jobs 状态机、progress 模型、drafts 断点续传、共享 TS 类型、健康检查——这十件事在这里定一次，各域契约（导入/提取/结构化/发布/工作台/主页）只引用、不重定义。
>
> **真源**：技术架构以飞书《技术方案 · 创作者中心与消费链路》为权威，本地落地以 `creator-builder/docs/01-详细技术方案.md` 为准；产品行为以 `docs/开工总纲-创作者中心主链路.md` 第五/八章为准；验收口径以 `docs/测试验收-创作者中心主链路.md` 为准。
>
> **三条硬规则（贯穿全文，每节标注落地点）**：
> 1. **永不裸转圈** —— 一切耗时步骤经 jobs + SSE 推「总进度 + 子任务清单 + 量化文案 + 边生成边显示」；连接即下发 `state_snapshot`；慢任务发 `slow_hint`/`field_stuck`；degraded 也给进度短语，绝无空转圈。落地于 §5 SSE 帧、§7 progress、§6 jobs。
> 2. **绝不裸露错误码** —— 对外只出 `ErrorEnvelope`（人话 `userMessage` + 可操作 action + retriable），内部 code/堆栈/HTTP 状态永不外泄。落地于 §3 错误信封。
> 3. **已生成内容不丢** —— 字段级/候选级逐项落库；中断/超时/取消保留已完成部分；草稿 + 断点续传回精确断点；刷新走 `state_snapshot` 恢复、不打回从头。落地于 §6 取消语义、§7 progress、§8 drafts。
>
> 本文只写契约（markdown + TS 类型片段 + SQL DDL），不写业务实现代码。TS 片段最终归集到 `src/shared/`（zod schema 即 OpenAPI 3.1 真源），DDL 归 `src/infra/pg/migrations/`。

## 1. 路由与版本约定

### 1.1 前缀与版本

- **统一前缀**：`/api/v1`。所有业务端点都在 `/api/v1` 下，无前缀的 `/health`、`/ready`、`/metrics` 例外（见 §10）。
- **版本策略**：URL path 版本（`/api/v1`）。v1 内的迁移只加不减、向后兼容（与 DB 迁移同口径）；破坏性变更才升 `/api/v2`，两版并存过渡。
- **OpenAPI 真源**：`src/shared/` 下 zod schema 即 OpenAPI 3.1 真源（B-07），前端 codegen TS client 由它生成；本文 TS 片段是其人读镜像，冲突以 zod schema 为准。

### 1.2 资源命名

- **资源用复数名词**，kebab 不用、用全小写下划线无需（路径段用复数英文名词，多词用连字符）：`/import/jobs`、`/snapshots`、`/capabilities`、`/capability-versions`、`/publish-batches`、`/creators`、`/notifications`、`/market/listings`。
- **子资源嵌在父资源下**：`/snapshots/{snapshotId}/extract`、`/capabilities/{capabilityId}/versions`、`/versions/{versionId}/structure`、`/versions/{versionId}/publish`、`/candidates/{candidateId}/retry`、`/jobs/{jobId}/events`、`/jobs/{jobId}/cancel`、`/publish-batches/{batchId}/items/{itemId}/retry`、`/creators/{creatorId}/profile`。
- **动作型子资源（非 CRUD 的领域动作）用动词子路径 + POST**：`.../extract`、`.../structure`、`.../publish`、`.../retry`、`.../cancel`、`/follows`、`/likes`。这些是「对资源执行的命令」，不是资源本身。
- **集合 GET 用复数返回 `Paginated<T>`，单体 GET 用 `{id}` 返回 `Envelope<T>`**（见 §2）。
- **manifest 寻址用不可变 version_id**（飞书第 12 条差异）：`GET /api/v1/market/manifests/{versionId}`，slug 仅展示、不用于寻址。

### 1.3 标识符（ID）约定

- **对外 ID 一律字符串**。主键用 UUID v7（时间有序，利于索引与 cursor 分页）。类型别名见 §9。
- **不可变业务 slug**（`capabilities.slug`）单独一类：URL 安全、创建后不可变、用于公开主页/市集展示路径 `/a/{slug}`，但 API 寻址仍用 UUID（slug 唯一约束保证可双向解析）。
- **匿名身份键** `consumerKey = hash(share_token + anon_cookie)`，仅计量/限流用，本期 usage 置空不写。

### 1.4 HTTP 方法

| 方法 | 语义 | 幂等性 |
|---|---|---|
| GET | 读，无副作用 | 天然幂等 |
| POST | 建资源 / 执行领域命令 | **非幂等，须带 `Idempotency-Key`（见 §4）** |
| PATCH | 局部更新（如软字段编辑） | 由 `If-Match`/版本号控制并发；published 后 PATCH manifest 强制建新版本 |
| DELETE | 软删 / 下架 | 幂等（重复 DELETE 同结果） |

PUT 不用（避免「整体替换」语义与软硬字段分层冲突）。

## 2. 统一响应包络与分页

### 2.1 成功包络

**取舍：用「轻包络」——成功响应统一裹一层 `{ data, meta? }`**，不裸返实体。理由：(1) 给 `meta` 留位放占位语义（usage 本期置空，需 `placeholder` 标注，见技术方案 B-32/B-33）、traceId、降级标记；(2) 列表/单体形态统一，前端 codegen 一致解包。

```jsonc
// 单体
{ "data": { /* T */ }, "meta": { "traceId": "..." } }
// 集合（见 2.3 分页）
{ "data": [ /* T[] */ ], "meta": { "traceId": "...", "page": { /* PageMeta */ } } }
```

`meta` 永远可选字段集合，不破坏向后兼容；新增 meta 字段不算破坏性变更。

### 2.2 占位语义（usage 本期置空的落地）

usage 类字段（本月消耗、调用量、活跃消费者、token 趋势、收益、作品墙调用次数等）**本期统一返回 `null` 值 + `placeholder` 文案**，不是空错误、不是 0 误导、不裸转圈：

```jsonc
{
  "data": {
    "publishedCount": 7,                          // 非 usage：真实值
    "monthlyInvocations": null,                    // usage：值为 null
    "activeConsumers": null
  },
  "meta": {
    "placeholders": {                              // 哪些字段是占位、占位文案是什么
      "monthlyInvocations": "暂无数据 / 上线后填充",
      "activeConsumers": "暂无数据 / 上线后填充"
    }
  }
}
```

前端见 `placeholders[field]` 即渲染占位态（灰条/得体文案），不渲染 0、不渲染错误、不转圈。各聚合域契约（工作台/主页）按此口径标注各自 usage 字段，不再重定义机制。

### 2.3 分页：cursor 唯一（不用 offset）

**取舍：统一 cursor 分页，全程不提供 offset 分页。** 理由：(1) 主要列表（jobs、candidates、capabilities、notifications、market listings）都按时间/UUID v7 有序、增长中，offset 在写入并发下会漏读/重读；(2) cursor 对「边生成边追加」的列表（候选逐个浮现）天然友好；(3) 单一分页风格降低各域契约与前端心智负担。

请求：`GET /api/v1/{collection}?cursor={opaqueCursor}&limit={n}&order=asc|desc`
- `cursor` 不透明字符串（服务端 base64 编码 `{sortKey, id}`），首页不传。
- `limit` 默认 20，上限 100。
- `order` 默认 `desc`（最新在前）；候选/字段这类「追加流」用 `asc`。

响应 `meta.page`：

```jsonc
"page": {
  "nextCursor": "eyJ...",   // null 表示到底
  "hasMore": true,
  "limit": 20,
  "order": "desc"
}
```

**不返回 total**（count(*) 在增长大表上昂贵且会随写入抖动）。需要计数的场景（如「已识别 3 / 9」）由 SSE 进度事件携带 `done/total`（见 §7），不靠分页 total。

## 3. 错误信封 ErrorEnvelope（绝不裸露错误码的落地）

### 3.1 形态

**所有非 2xx 响应、所有 SSE `error` 帧、所有前端可见失败，统一只出 `ErrorEnvelope`。** 内部 code 是内部枚举（用于日志/告警/前端文案映射，不是 HTTP 状态、不是堆栈、不是英文报错），`userMessage`（权威字段名，见 §11.B）是已翻成人话的中文、可直接渲染。

> **收紧口径见 §11.B（Codex#11 绝不裸露错误码）**：`error.code` **仅供日志/告警/文案映射，UI 永不渲染**；唯一可对 UI 展示的是 `userMessage`（人话）+ `action`（核心三类 `retry|change_input|escalate`）；`traceId` 可作「反馈代码」展示但非错误码。登录失败重定向用 opaque `failureId`、不带内部 code。**字段名权威且唯一为 `userMessage`，全契约不再有 `message` 同义字段。**

```jsonc
{
  "error": {
    "code": "EXTRACT_UPSTREAM_TIMEOUT",   // 内部枚举（见 3.3 分类表），稳定、可映射文案/告警
    "userMessage": "提取暂时没能完成，请稍后重试。", // 对外人话，可直接渲染；绝不含 500/堆栈/英文报错
    "retriable": true,                      // 是否值得自动/手动重试
    "action": "retry",                      // 给用户的可操作退路（见 3.2）
    "traceId": "01J...",                    // 关联日志/Sentry，前端可展示「反馈代码」但不当错误码
    "details": {                            // 可选：结构化补充，供前端做更精细 UI（绝不放堆栈）
      "stuckAt": "段 5 / 9",               // 例：人话定位（对应「上游解析中断 · 段 5/9」）
      "fields": ["tagline"]                // 例：哪些字段失败
    }
  }
}
```

**硬约束（CI/评审守门）**：
- `userMessage` 永远是面向创作者的中文人话，**禁止**出现 HTTP 状态码、`Error:`、stack trace、异常类名、英文原始报错、SQL/Redis/驱动报错串。
- HTTP 状态码只在响应行/SSE 不裸进 body 文案；`code` 是内部枚举不是状态码。
- `details` 只放结构化、可安全展示的补充（人话定位、字段名、可重试项 id），**禁止**放堆栈/原始 message/内部路径。
- LLM/外部调用失败重试 ≤ 2 后才落终态错误信封（与 LLM Gateway 重试策略一致）。

### 3.2 action 枚举（用户的可操作退路）

| action | 含义 | 前端表现 |
|---|---|---|
| `retry` | 同样输入重试即可能成功（瞬时/上游抖动） | 「重试」按钮，点击重发原请求 |
| `change_input` | 需改输入再试（输入不合法/内容不足/被去敏后空） | 「改输入」/「返回上一步」，引导回到可编辑处 |
| `escalate` | 自助无解，需转人工/反馈 | 「转人工」/「反馈」，带 traceId |
| `wait` | 后台仍在跑，稍候自动恢复（慢任务、可后台继续） | 「再等等 / 转后台」，配 `slow_hint`；不转圈，给进度短语 |
| `none` | 无需动作（如纯占位 / 信息态） | 仅展示说明，无动作按钮 |

> 结构化「卡住三退路」（用已生成继续 / 只重生成卡住字段 / 再等等）是 `field_stuck` SSE 帧 + 这里 action 的组合：再等等=`wait`、只重生成=`retry`(scoped)、用已生成继续=前端动作不回服务端。同一处重试两次仍失败 → 落 `action:'escalate'`（重试/改输入/转人工三选一）。

### 3.3 错误分类总表（HTTP 状态 + 业务 code）

> 列：HTTP 状态 → 典型内部 code（枚举，可扩展，命名 `{DOMAIN}_{REASON}`）→ retriable/action 缺省 → 人话 `userMessage` 模板。各域契约在此基础上扩 code，但 **action/retriable 缺省须遵此表**，`userMessage` 须人话。

| HTTP | 内部 code（示例枚举） | retriable | action | 人话 `userMessage` 模板 |
|---|---|---|---|---|
| 400 | `VALIDATION_FAILED` | false | `change_input` | 「输入有点问题：{字段人话}，改一下再试。」 |
| 400 | `INPUT_TOO_SMALL` / `IMPORT_NO_CONTENT` | false | `change_input` | 「这次没读到可用内容，换个目录/文件再导入。」 |
| 401 | `UNAUTHENTICATED` | false | `escalate` | 「登录态失效了，请重新登录。」（前端跳登录） |
| 403 | `FORBIDDEN` | false | `escalate` | 「你没有权限做这个操作。」 |
| 404 | `NOT_FOUND` | false | `change_input` | 「没找到对应内容，可能已被删除或链接失效。」 |
| 409 | `IDEMPOTENCY_CONFLICT` | false | `none` | （见 §4，通常对前端透明，返回首次结果或冲突说明） |
| 409 | `STATE_CONFLICT` | false | `change_input` | 「当前状态不支持这个操作（如已发布需建新版本）。」 |
| 409 | `ALREADY_PUBLISHED` | false | `none` | 「这个能力已发布过了，无需重复发布。」 |
| 422 | `STRUCTURE_FIELD_FAILED` | true | `retry` | 「这个字段没生成出来，可重试或改输入。」 |
| 423 | `RESOURCE_LOCKED` | true | `wait` | 「这条任务正在被处理，请稍候。」 |
| 429 | `RATE_LIMITED` | true | `wait` | 「请求有点频繁，歇一下再试。」（带 Retry-After） |
| 499 | `CLIENT_CANCELLED` | false | `none` | （客户端主动取消，通常不展示错误） |
| 500 | `INTERNAL` | true | `retry` | 「服务开小差了，请重试。」（绝不带堆栈/500 字样进 `userMessage`） |
| 502 | `LLM_UPSTREAM_FAILED` / `EXTRACT_UPSTREAM_TIMEOUT` | true | `retry` | 「上游处理暂时不稳定，请稍后重试。」 |
| 503 | `DEPENDENCY_UNAVAILABLE` | true | `wait` | 「系统正在恢复，请稍候再试。」 |
| 504 | `JOB_TIMEOUT` | true | `retry` | 「这一步超时了，可重试或稍后再看。」 |

> AI/外部 degraded（OpenRouter 不稳）**不算依赖失败、不停服**（见 §10）：调用方返回 `LLM_UPSTREAM_FAILED` + `action:'retry'`/`'wait'` + 进度短语，绝不裸转圈、绝不裸 502。

### 3.4 traceId

每个请求生成 `traceId`（UUID v7 / ULID），贯穿日志、Sentry、outbox、SSE。成功响应放 `meta.traceId`，错误放 `error.traceId`。前端可作为「反馈代码」展示给用户，但它不是错误码、不替代人话 `userMessage`。

## 4. 幂等约定（B-09）

### 4.1 Idempotency-Key 头

- **所有写命令（POST / PATCH / DELETE）必须带 `Idempotency-Key` 头**（建任务、建能力、发布、批量发布每 item、重试、取消 Job、配对铸码、标已读、社交动作 follow/like 及其 DELETE 取消）。客户端生成（UUID v7），同一逻辑操作复用同一 key。**DELETE 不因「天然幂等」而豁免**——仍带 key + 固定 scope，重复请求回放首次结果（行为矩阵 §4.2），与所有其他写操作口径一致。GET/读不需要。**唯一豁免（已显式标注、有理由）**：`POST /auth/logout`（会话销毁、无产物、无连坐，§10-auth）；以及**不写库的「带请求体只读」POST**——`POST /import/uploads/presign`、`POST /versions/{versionId}/market-card/preview`（只签 URL / 只算预览、不创建任何持久化行，与 GET 同语义，故非写命令、key 可选）。**GET 回调例外**：`GET /api/v1/auth/callback`（OIDC browser redirect、code/state 一次性，§10-auth）本就是 GET，不属于 POST/PATCH/DELETE 写命令 `Idempotency-Key` 体系，单列于此说明其一次性由 OAuth code/state 语义自带，无需 key。
- **作用范围 scope**：`(scope, key)` 唯一。scope = 端点逻辑名（如 `import.create`、`publish.version`、`publish_batch.item`）。**批量发布每 item 用独立 key**（无连坐，见技术方案决策⑤）。
- **判定在 app 层**，落 `idempotency_keys` 表，含 `request_hash`（防 key 复用但 body 不同）、`response_ref`（首次成功结果引用，重放直接回放）、租约 `locked_at/expires_at`（防并发重入）。

### 4.2 行为矩阵

| 情形 | 行为 | 返回 |
|---|---|---|
| key 首次出现 | 取租约 → 执行 → 落 response_ref | 正常 2xx |
| key 重复 + request_hash 相同 + 首次已完成 | 不重复执行，回放首次结果 | 首次的 2xx（**对前端透明**，双击/刷新/重连不产生重复任务/重复发布） |
| key 重复 + request_hash 相同 + 首次仍在租约中 | 不重入 | `423 RESOURCE_LOCKED` + `action:'wait'`（前端稍候重试同 key） |
| key 重复 + request_hash **不同** | 拒绝（key 被复用于不同请求） | `409 IDEMPOTENCY_CONFLICT` |

> 对应验收：选择结构化-08（重复点/刷新只发一次）、贯穿-27（双标签页不重复发布）、导入重复点只跑一次。BullMQ `jobId` 去重是第二道闸（同一 job 不重复入队），与 app 层幂等键叠加。

```sql
-- idempotency_keys（B-09）
CREATE TABLE idempotency_keys (
  scope         text        NOT NULL,           -- 端点逻辑名
  key           text        NOT NULL,           -- 客户端 Idempotency-Key
  request_hash  text        NOT NULL,           -- hash(规范化 body)，防 key 复用 body 异
  response_ref  jsonb,                          -- 首次成功结果引用（回放用）
  status        text        NOT NULL DEFAULT 'locked', -- locked|completed|failed
  locked_at     timestamptz NOT NULL DEFAULT now(),
  expires_at    timestamptz NOT NULL,           -- 租约到期（防死锁）
  created_at    timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (scope, key)
);
CREATE INDEX idx_idem_expires ON idempotency_keys (expires_at) WHERE status = 'locked';
```

## 5. SSE 帧协议（永不裸转圈的核心机制，B-12）

### 5.1 连接

- 端点：`GET /api/v1/jobs/{jobId}/events`（job 流）；结构化字段流与 job 流复用同端点按 `kind` 区分，或 `GET /api/v1/versions/{versionId}/structure/events`（structure 流）；session 流为 Runtime 预留（B-40，本期仅冻结 schema、无可调用端点）。
- **鉴权见 §11.C（Codex#5 SSE 鉴权统一）**：所有 SSE 流**统一用同源 Cookie 会话鉴权，禁 query-string token**；鉴权/权限失败必须在**建流之前**返 HTTP `ErrorEnvelope`（`401`/`403`），**不得**用 SSE `error` 帧表达鉴权失败（`error` 帧只表达已建流后的业务失败终态）。
- 媒体类型 `text/event-stream`；HTTP/1.1、关闭代理缓冲、长读超时（nginx 透传，见技术方案 §6.1）。
- 事件源 = Redis Streams `events:{kind}:{id}`（worker `XADD`，api SSE Hub 桥接 + 补发 + 跟流）。

### 5.2 连接即下发 state_snapshot（硬规则①落地）

**连接建立（含断线重连超窗）后，服务端的第一帧永远是 `state_snapshot`**，把当前全量状态一次推给前端，前端用它重置 UI 再续增量。这是「刷新/重连不打回从头、已生成不丢」的机制基座。snapshot 按流类型分三型：

| 流类型 kind | state_snapshot 内容 | 对应 |
|---|---|---|
| `job` | `jobs.progress` 全量（总进度 + 子任务清单 + 量化文案 + 已追加 items 摘要） | 导入/提取/批量发布进度恢复 |
| `structure` | `structure_state` 全量（每软字段状态/已生成值/卡住时长 + 硬字段锁定值） | 结构化字段恢复、已生成字段不丢 |
| `session` | 会话视图全量（Runtime 预留，本期 schema-only） | B-40，本期不交付 |

### 5.3 事件类型集合

> 每帧 SSE 标准格式：`id:`（= Redis Stream entry id，用于 Last-Event-ID）、`event:`（类型名）、`data:`（JSON）。所有 `data` 是 `SSEFrame` 的 `payload`。

| event | 何时发 | payload 要点 | 硬规则 |
|---|---|---|---|
| `state_snapshot` | 连接首帧 / 重连超窗 | 见 5.2 三型全量 | ①③ 不从头、不丢 |
| `progress` | 总进度推进 | `{ percent, phrase, done, total, unit }` 量化文案 | ① 不裸转圈 |
| `subtask` | 子任务状态变化 | `{ subtasks: SubtaskView[] }` 或单条 `{ key, status }`（pending→running→done/failed） | ① 子任务清单点亮 |
| `item-appended` | 边生成边显示追加一项 | `{ item: <候选/段/批量item 摘要> }` | ①③ 逐个浮现、已生成不丢 |
| `field_start` | 软字段开始生成 | `{ field, index, total }` | ① |
| `field_delta` | 软字段流式增量 | `{ field, deltaText }` | ① 一项项蹦出来 |
| `field_done` | 软字段生成完成 | `{ field, value }`（已落库） | ③ 字段级落库不丢 |
| `field_stuck` | 软字段生成偏慢 | `{ field, elapsedMs, options:['continue','regen','wait'] }` | ① 三退路、不干等 |
| `slow_hint` | 整体任务偏慢 | `{ phrase, elapsedMs }` 进度短语 | ① degraded 也给短语 |
| `error` | 失败终态 | `ErrorEnvelope`（见 §3，整体到 done） | ② 不裸错误码 |
| `done` | 任务终止（成功/失败/取消统一终止帧） | `{ status: JobStatus, result?, error? }` | 终止信号，前端关流 |
| `heartbeat` | 周期保活（默认 15s） | `{ ts }`（亦可用 SSE comment `:hb`） | ① 连接活着的证据 |

> 注：`item-appended` 用连字符（与 PRD 验收「逐个浮现」措辞对齐）；`field_*` 用下划线（与技术方案 B-25 `field_start/field_delta/field_done/field_stuck` 命名一致）。两套命名各自稳定，前端按 `event` 名分发。

### 5.4 断线重连 / Last-Event-ID

- 每帧 `id:` = Redis Stream entry id（单调递增）。
- 客户端用 `fetch-event-source`，重连自动带 `Last-Event-ID` 头。
- 服务端补发逻辑：
  - **Last-Event-ID 仍在 Stream 窗口内** → 从该 id 之后补发增量（不重推 snapshot）。
  - **超窗（id 已被裁剪）或无 Last-Event-ID** → 先推 `state_snapshot`（按 kind 三型）重置，再续流。
- 对应验收：导入-25（终端跑完网页自动接上）、提取-23（刷新已识别不丢）、贯穿-22（断线续传到真实状态）、贯穿-28（结构化只补未生成字段）。

### 5.5 心跳与终止

- **心跳**：默认每 15s 发 `heartbeat`（或 SSE comment `:hb\n\n`），防中间层断连、给前端「连接活着」信号。
- **终止事件**：任务到终态（completed/failed/cancelled）发一帧 `done`（失败时同时已发过 `error`），前端收到 `done` 即关闭 EventSource、停止重连。无 `done` 的断开视为网络中断 → 触发重连流程。

## 6. jobs 状态机 + 执行租约/fencing（B-10 / B-11）

### 6.1 状态机

```
            ┌──────────────► cancelled  (取消，换 fence；保留已完成产物)
            │
queued ──► running ──► completed
            │
            └──────────► failed  (重试 ≤2 后落终态错误信封)
```

- **PG `jobs` 表是任务状态唯一真源**，BullMQ 只触发执行（`jobId` 去重）。
- `running → completed/failed/cancelled` 是终态；终态不可逆（重试/重入队是新 `attempt_no` + 新 `fence_token`，不是回 running 原 attempt）。
- 取消语义（B-11）：标 `cancelled` **并换 `fence_token`** → 被取消的旧执行因 fence 不匹配再也无法回写 → BullMQ remove → 控制流通知 worker 安全点停 → **已完成产物保留**（硬规则③）。

### 6.2 执行租约 / fencing（防重入、防旧覆盖新）

| 字段 | 作用 |
|---|---|
| `attempt_no` | 第几次执行（sweeper 重入队才递增） |
| `lease_owner` | 当前持租 worker 实例标识 |
| `lease_until` | 租约到期时刻（worker 周期续期） |
| `fence_token` | 单调令牌，每次领取/重入队/取消都换新 |

**铁律（worker 写入必须遵守）**：所有对 `jobs` 及其产物（candidates / 字段 / 快照 / 批量 item）的写入都带 `WHERE job_id = ? AND fence_token = ?`。fence 不匹配 → 写入影响 0 行 → 旧 worker 安全退出，杜绝「旧执行覆盖新执行 / 重复候选 / 重复字段」。

> **写法必须遵 §11.A「受保护写入规范模式」**（Codex#3）：fence 校验**内联进单条事务 CTE 的数据源**（`... SELECT ... FROM jobs WHERE id=:jobId AND fence_token=:fence AND status='running'`），**禁止**「先 SELECT 校验、再独立 INSERT/UPDATE」两步写法（有 TOCTOU 窗口）。可复制 SQL 模板见 §11.A。

**sweeper 重入队条件**：仅当 `lease_until` 过期（worker 死/卡），sweeper 才 `attempt_no += 1` + 换 `fence_token` 重入队。lease 未过期不抢。对应技术方案 B-16。

### 6.3 jobs 表 DDL

```sql
CREATE TABLE jobs (
  id            uuid        PRIMARY KEY DEFAULT gen_uuid_v7(),
  type          text        NOT NULL,           -- import|extract|structure|publish_batch|evaluate|runtime_gen
                                                 -- 本期实际注册 processor 仅前四类；后两类 schema 冻结、不注册
  status        text        NOT NULL DEFAULT 'queued', -- queued|running|completed|failed|cancelled
  owner_user_id uuid        NOT NULL REFERENCES users(id),
  subject_ref   jsonb,                          -- 任务作用对象（snapshotId/versionId/batchId 等）
  progress      jsonb       NOT NULL DEFAULT '{}'::jsonb, -- ProgressView（见 §7），state_snapshot(job) 真源
  result        jsonb,                          -- 成功产物引用
  error         jsonb,                          -- 失败时 ErrorEnvelope.error（人话，非堆栈）
  -- 执行租约 / fencing
  attempt_no    int         NOT NULL DEFAULT 0,
  lease_owner   text,
  lease_until   timestamptz,
  fence_token   bigint      NOT NULL DEFAULT 0, -- 领取/重入队/取消递增换新
  -- 时间戳
  created_at    timestamptz NOT NULL DEFAULT now(),
  started_at    timestamptz,
  updated_at    timestamptz NOT NULL DEFAULT now(),
  finished_at   timestamptz
);
CREATE INDEX idx_jobs_owner_status ON jobs (owner_user_id, status, created_at DESC);
CREATE INDEX idx_jobs_lease ON jobs (status, lease_until) WHERE status = 'running'; -- sweeper 对账
CREATE INDEX idx_jobs_type_status ON jobs (type, status);
```

> worker 续期/写入示例（说明 fencing 用法，非实现代码）：
> `UPDATE jobs SET progress=$p, updated_at=now() WHERE id=$jobId AND fence_token=$myFence;`
> 影响 0 行 → 自己已被 fence out → 安全退出。

## 7. progress 模型（总进度 + 子任务 + 量化文案 + 边生成边显示）

`jobs.progress` 持久化 + 作为 `state_snapshot(job)` 全量真源；SSE `progress`/`subtask`/`item-appended` 是它的增量推送。形态：

```jsonc
{
  "percent": 68,                                 // 总进度 0-100，单调不倒退
  "phrase": "68% · 已抓取 146 / 215 段会话 · 5,210 / 8,420 条消息", // 量化文案（导入-07/10）
  "done": 146, "total": 215, "unit": "段会话",   // 量化分子分母（不靠分页 total）
  "subtasks": [                                   // 子任务清单，依次点亮
    { "key": "credential", "label": "连接凭证",            "status": "done" },
    { "key": "fetch_index", "label": "拉取会话索引",        "status": "done" },
    { "key": "redact",      "label": "导入消息并抹掉隐私信息","status": "running" },
    { "key": "segment",     "label": "切分成段落",          "status": "pending" },
    { "key": "snapshot",    "label": "生成原始数据",        "status": "pending" }
  ],
  "items": [ /* 边生成边显示已追加项的摘要：候选/段/批量item */ ],
  "slow": false                                   // 偏慢标记，配 slow_hint 帧
}
```

**子任务清单按流类型有标准序**（验收逐条点亮）：
- **导入**（导入-08 / 贯穿-01，五项）：连接凭证 → 拉取会话索引 → 导入消息并抹掉隐私信息 → 切分成段落 → 生成原始数据。
- **提取**（提取-03，五项）：分析会话段落 → 聚类相似工作流 → 形成候选能力 → 评估频率与可打包度 → 按成功率排序。
- **结构化**：以字段流（`field_*`）+ `structure_state` 表达，subtasks 表「正在补全字段 4 / 6」类进度（贯穿-07）。

`SubtaskStatus = 'pending' | 'running' | 'done' | 'failed'`。`percent`/`done` 单调不倒退（导入-10）；`total` 确定后稳定。

## 8. drafts / 断点续传状态机（F-15 / 外壳首页-33 / 贯穿-15）

### 8.1 模型

五步任一步可存草稿；草稿记录**所处步骤 + 该步进度**，续传回精确断点、已生成内容不丢。两份真源协作：
- **步骤级**：`drafts` 表（草稿条数据源，工作台展示「结构化中 60%」、多条可区分）。
- **字段级**：`capability_versions.structure_state`（结构化每字段已生成值/状态/卡住时长，断点续传精度靠它，贯穿-28）。

### 8.2 step 枚举

```
import → extract → select → structure → publish
```

`select` 是纯前端步（不调模型不写库），其草稿仅记「已选 candidateId / 全部发布」于 draft payload，不产生服务端任务。

### 8.3 草稿状态机

```
active ──► (推进各步) ──► completed (走到发布成功)
   │
   ├──► abandoned (用户主动放弃)
   └──► (无操作过期归档，可配)
```

### 8.4 drafts 表 DDL（基表 + 后置 ALTER 闭合跨域 FK，Codex#18-r4）

> **建表顺序闭合（Codex#18-r4）**：`drafts` 是「核心」表（紧随 `users`/`jobs` 建），但其落点引用列指向**后建的导入/结构化/发布域表**（`raw_snapshots`/`capability_versions`/`publish_batches`），同时导入域 `import_pairings.draft_id` 又反向引用 `drafts`——内联这些跨域 FK 会形成「建表顺序环」（核心表前向引用后建表）。**沿用冻结表已用的「先建基表 + 后置 ALTER ADD CONSTRAINT」模式**：基表只内联**指向已建核心表**的 FK（`owner_user_id→users`、`extract_job_id→jobs`，二者均在 `drafts` 之前建），其余 4 条跨域 FK（`snapshot_id`/`version_id`/`batch_id`/`import_pairings.draft_id`）放到统一的**后置 ALTER 阶段**（与 70 §9.4 冻结表 ALTER 同阶段，见 §11.G 后置 FK 闭合清单）。

```sql
-- 基表：仅本表必需列 + PK + 指向已建核心表（users/jobs）的内联 FK；跨域 FK 后置（见下方 ALTER）
CREATE TABLE drafts (
  id              uuid        PRIMARY KEY DEFAULT gen_uuid_v7(),
  owner_user_id   uuid        NOT NULL REFERENCES users(id),  -- 核心表，drafts 之前已建，可内联
  status          text        NOT NULL DEFAULT 'active', -- active|completed|abandoned
  current_step    text        NOT NULL DEFAULT 'import', -- import|extract|select|structure|publish
  step_progress   jsonb       NOT NULL DEFAULT '{}'::jsonb, -- { percent, phrase }，草稿条展示「结构化中 60%」
  -- 各步落点引用（续传回精确断点）。指向后建域表的 FK 全部后置（§11.G），此处仅声明列、不内联跨域 FK。
  snapshot_id     uuid,                                      -- import 产物 → raw_snapshots(id)（后置 FK fk_drafts_snapshot）
  extract_job_id  uuid        REFERENCES jobs(id),           -- extract 落点（jobs 为核心表、已建，可内联）
  selection       jsonb,                                     -- select：选定 candidateId / 全部发布
  version_id      uuid,                                      -- structure/publish 落点 → capability_versions(id)（后置 FK fk_drafts_version）
  batch_id        uuid,                                      -- 批量发布落点 → publish_batches(id)（后置 FK fk_drafts_batch，Codex#18-r4 补 FK）
  title           text,                                      -- 草稿条可读标题（区分多条）
  created_at      timestamptz NOT NULL DEFAULT now(),
  updated_at      timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX idx_drafts_owner_active ON drafts (owner_user_id, status, updated_at DESC);
```

> **后置 ALTER（在 `raw_snapshots`/`capability_versions`/`publish_batches` 与 `import_pairings` 均已建后执行，约束名固定见 §11.G）**：
>
> ```sql
> -- drafts 落点跨域 FK（指向导入/结构化/发布域表，建表顺序晚于 drafts，故后置）
> ALTER TABLE drafts
>   ADD CONSTRAINT fk_drafts_snapshot FOREIGN KEY (snapshot_id)  REFERENCES raw_snapshots(id),
>   ADD CONSTRAINT fk_drafts_version  FOREIGN KEY (version_id)   REFERENCES capability_versions(id),
>   ADD CONSTRAINT fk_drafts_batch    FOREIGN KEY (batch_id)     REFERENCES publish_batches(id); -- Codex#18-r4 补齐缺失 FK
> -- import_pairings 反向引用 drafts（导入域内联会与 drafts 跨域 FK 形成环，故 draft_id FK 一并后置，见 20 §6.4）
> ALTER TABLE import_pairings
>   ADD CONSTRAINT fk_pairings_draft  FOREIGN KEY (draft_id)     REFERENCES drafts(id);
> ```

> 续传：草稿条 → 读 `current_step` + 对应落点引用 → 回到该步 → 该步内细粒度恢复走 SSE `state_snapshot`（job/structure），已生成内容原样回显（外壳首页-30/33、贯穿-10/16）。多条草稿按 `owner_user_id` 各自独立、互不串台（外壳首页-34）。

  selection?: unknown;
  versionId?: VersionId;
  batchId?: BatchId;
  createdAt: IsoDateTime;
  updatedAt: IsoDateTime;
}

// ---------- 健康检查 ----------
export type HealthStatus = 'ok' | 'degraded' | 'down';
export interface DependencyHealth {
  name: 'db' | 'redis_queue' | 'redis_hot' | 'minio' | 'logto' | 'llm';

### 10.2 真失败 vs degraded

## 11. 跨域共识硬规则（Codex 对抗结论 → 唯一权威裁定）

> 本节把 Codex 跨文件评审（`creator-builder/.reviews/phase1-codex-r1.txt`）指出的、必须全域口径一致的问题，在此定为**唯一权威规则**。各域契约（10/20/30/40/50/60/70）**只引用本节、不得各自重述或偏移**；与本节冲突的旧文以本节为准。每条标注 Codex 编号便于回溯。

### 11.A 受保护写入规范模式（Codex#3 · fence 写入闭环）

**问题**：fence 租约此前只停留在「前置检查示例」（§6.2 铁律是 WHERE 条件，但落库表给的多是「先 SELECT 校验、再 INSERT/UPDATE」的两步示例），存在 TOCTOU 窗口——旧 worker 检查通过后、写入前被 fence out，仍可能写脏。

**裁定（全域强制）**：所有 worker / sweeper 对 **job 结果、job 进度、以及任何带 job 血缘的产物**（candidates / 字段 / 快照段 / 批量 item / structure_state / manifest 等）的写入，**必须是单条事务内的 CTE，把 fence 校验内联进同一条 SQL 的数据源**，杜绝「先查后写」两步。校验三要素恒定：`id = :jobId AND fence_token = :fence AND status = 'running'`。fence 不匹配或已离开 running → `SELECT` 命中 0 行 → 写入影响 0 行 → 旧 worker 据「affected = 0」安全退出。**禁止**任何「先 `SELECT fence_token` 校验、再独立 `INSERT/UPDATE`」的两步写法。

**可复制 SQL 模板（受保护写入，复制即用）**：

```sql
-- 模板 1 · 受保护 UPDATE（写 job 自身：进度 / 结果 / 终态）
WITH guard AS (
  SELECT id, fence_token
  FROM jobs
  WHERE id = :jobId
    AND fence_token = :fence
    AND status = 'running'
  FOR UPDATE                       -- 防与 sweeper 取消/重入队竞争
)
UPDATE jobs j
SET progress = :progress, updated_at = now()
FROM guard
WHERE j.id = guard.id;            -- 仅 guard 命中才更新；0 行 = 已被 fence out
-- 应用层：rowCount = 0 ⇒ 不是错误，是「我已不是当前执行」，安全退出本 attempt。

-- 模板 2 · 受保护 INSERT（写带 job 血缘的产物：候选 / 段 / 批量 item / 证据 …）
INSERT INTO capability_candidates (id, owner_user_id, extract_job_id, /* … */)
SELECT gen_uuid_v7(), :ownerUserId, j.id, /* … */
FROM jobs j
WHERE j.id = :jobId
  AND j.fence_token = :fence
  AND j.status = 'running'
ON CONFLICT (extract_job_id, slug) DO NOTHING;  -- fence 守门 + 业务去重键叠加
-- 产物表自身带 job 血缘列（如 extract_job_id），既支撑 fence 守门又支撑取消后归属判定。

-- 模板 3 · 受保护 UPDATE（写产物：如结构化 structure_state / manifest，fence 经 jobs 联表）
UPDATE capability_versions v
SET structure_state = :state, updated_at = now()
FROM jobs j
WHERE j.id = :jobId
  AND j.fence_token = :fence
  AND j.status = 'running'
  AND v.id = :versionId;          -- 产物经 job 联表校验 fence；0 行 = 安全退出
```

**约束声明（各域必须遵守）**：
- 凡 worker/sweeper 写入，**必须**采用上述模板之一（CTE / `... SELECT ... FROM jobs WHERE id=:jobId AND fence_token=:fence AND status='running'`），**不得**用两步「查 + 写」。
- 产物表**必须**带可被守门的 job 血缘列（`extract_job_id` / 经 `version_id`→job 的可联路径等），无血缘列不能享受 fence 守门。
- `rowCount = 0` 是**正常控制流**（「我已被 fence out」），不是异常、不落 ErrorEnvelope、不重试；worker 据此干净退出当前 attempt。
- 引用方：20（导入 worker 写 raw_snapshots/session_segments）、30（萃取 worker 写 candidates/evidence）、40（结构化 worker 写 structure_state/manifest）、50（批量发布 worker 写 batch item）、70（sweeper 重入队/补投）——全部遵 11.A。

### 11.B ErrorEnvelope 收紧：内部 code 仅日志，userMessage 唯一可展示（Codex#11 · 绝不裸露错误码）

**问题**：`error.code` 此前虽是「内部枚举」，但契约未**显式禁止** UI 渲染 code；登录失败回调把内部 code 直接拼进 `/login?error=<code>` URL（10-auth §3.2），等于把内部码暴给前端 / 进浏览器历史 / referer。

**裁定（全域强制，收紧 §3 与 §9 ErrorEnvelope 语义）**：
- **`error.code` 仅用于服务端日志 / 告警 / 文案映射查表，永不作为「展示内容」**。它不是给人看的，是给运维和文案映射表看的。
- **唯一可对 UI 展示的字段是 `error.userMessage`（人话）+ `error.action`（`retry|change_input|escalate`，本规则收敛为这三类可操作退路；`wait|none` 仍合法但属「信息态/后台态」，非「让用户改正错误」的 action）**。`traceId` 可作为「反馈代码」展示（仅用于用户向客服报障），但它**不是错误码、不替代人话**。
- **契约明确禁止**：UI / 前端**不得**渲染 `error.code`、HTTP 状态码、堆栈、异常类名、英文原始报错、SQL/驱动报错串。`details` 仅放结构化可安全展示项（人话定位、字段名、可重试项 id），禁放堆栈/原始 message/内部路径。
- **字段命名统一**：对外人话字段权威名**唯一为 `userMessage`**；§3.1 / §9 的共享 TS 类型、JSON 示例、错误分类表已全部用 `userMessage`，**全契约不再保留 `message` 同义字段**（含「人读镜像」说法一并删除）。各域错误用例表「人话」列即 `userMessage`，语义 = 「唯一可直接渲染的人话」。zod schema 真源以 `userMessage` 为准。
- **登录失败重定向用 opaque `failureId`，不带内部错误码**：10-auth 的失败回调**禁止** `/login?error=<内部code>`；改为服务端生成不透明 `failureId`（随机短串，服务端映射到内部 code + traceId 落日志），重定向 `/login?failureId=<opaque>`；前端拿 `failureId` 调一个「失败说明」读接口或直接据预置文案表渲染人话，**内部 code 永不进 URL / 前端**。

**收紧后的 ErrorEnvelope（权威形态，覆盖 §3.1 / §9）**：

```typescript
export type ErrorAction = 'retry' | 'change_input' | 'escalate' | 'wait' | 'none';
// 「让用户改正/重试」的三类核心退路：retry | change_input | escalate（Codex#11 收敛口径）。
// wait/none 为后台态/信息态，仍合法但不属「错误纠正动作」。
export interface ErrorEnvelope {
  error: {
    code: string;            // 内部枚举 {DOMAIN}_{REASON}。仅日志/告警/文案映射；UI 永不渲染
    userMessage: string;     // 唯一可对 UI 展示的人话（中文）；绝不含 code/状态码/堆栈/英文报错
    action: ErrorAction;     // 可展示：决定 UI 退路按钮
    retriable: boolean;
    traceId: TraceId;        // 可作「反馈代码」展示给用户报障；非错误码、不替代 userMessage
    failureId?: string;      // 仅登录等重定向场景：不透明失败标识，替代 URL 里的内部 code
    details?: Record<string, unknown>; // 结构化可安全展示项；禁放堆栈/原始报错/内部路径
  };
}
```

**约束声明**：CI/评审守门——前端代码出现「渲染 `error.code`」「`?error=<code>`」「把状态码/堆栈拼进可见文案」「`error.message` 字段引用」即判违规。各域错误用例表的「人话」列即 `userMessage`。引用方：10（登录失败 failureId）、全域（任何对外失败只出收紧后 ErrorEnvelope）。

### 11.C SSE 鉴权统一：同源 Cookie，建流前 HTTP 失败（Codex#5）

**问题**：40-structure §4.D 写「Bearer，SSE 经查询参/头透传 token」「连接级失败走 error 帧」，与 10-auth §5「Cookie only、禁 query token、鉴权失败前置 HTTP ErrorEnvelope」直接冲突。

**裁定（全域唯一权威，覆盖所有 SSE 流——job / structure / 未来 session）**：
- **统一用同源 Cookie 会话鉴权**（与决策「Logto 自托管 · 同源 Cookie 会话」一致）。`EventSource` / `fetch-event-source` 自动携带同源会话 Cookie；中间件按 §4 / 10-auth §4 校验后才建流。
- **禁止 query-string token**（token 会进访问日志 / referer / 浏览器历史，泄漏面大）；**禁止**自定义 header token 作为 SSE 主鉴权（EventSource 原生不支持自定义头，且与 Cookie 口径分叉）。SSE 鉴权口径**只有 Cookie 一种**。
- **鉴权 / 权限失败必须在「建流之前」以普通 HTTP 响应返 `ErrorEnvelope`**（未登录 `401 UNAUTHENTICATED` + `action:'escalate'`；非 owner / 缺角色 `403 FORBIDDEN` + `action:'escalate'`）。**绝不**用 SSE `error` 帧表达鉴权 / 权限失败——SSE `error` 帧只表达「已建流、已鉴权后的业务失败终态」。
- **owner 校验**同口径：流对应 job/version 必须属当前 `ctx.userId`，否则建流前 `403`。
- **流中途 token 过期**不强断已建立的流（连接已鉴权）；下次重连握手失效即握手期 `401`（HTTP，非帧），前端据此跳登录。重连握手失败发生在帧流之前，不破坏「先 `state_snapshot` 再续增量」恢复。

**约束声明**：40-structure §4.D 的「Bearer / 查询参 / 头透传 token / 连接级失败走 error 帧」表述**作废**，以 11.C 为准（须改齐 Cookie + 建流前 HTTP 失败）。引用方:40(structure SSE)、20/30/50(job SSE)、10(提供中间件)、所有未来 SSE 端点。

### 11.D outbox 连续安全前缀算法（Codex#1 · 70 域唯一权威算法）

**问题**：70-events §3.2 第 2 步把 `xid < :xmin` 写进 SQL `WHERE` 过滤。这会**永远遇不到「首条 unsafe 行」**——unsafe 行被 SQL 直接滤掉，应用层第 3 步「遇首条 `xid >= xmin` 即停」永不触发，于是 cursor 会越过被滤掉的低 seq unsafe 行，等它提交后已无人回头取它 → **事件永久漏消费**。

**裁定（70 及任何顺序消费者引用的唯一权威算法）**：**水位裁剪在应用层做，不在 SQL 过滤 `xid`。**

1. 取已提交水位：`xmin = pg_snapshot_xmin(pg_current_snapshot())`（所有 `xid < xmin` 的事务已确定提交/回滚）。
2. **按 `seq > :cursorSeq ORDER BY seq ASC LIMIT :batch` 取连续批**——**SQL 里不带任何 `xid` 过滤**，把可能 unsafe 的行也读进来。
3. 应用层**顺序扫描**该批，逐条比对 `xid` 与 `xmin`：
   - `xid < xmin` → 安全，处理它、记入「可提交的连续安全前缀」。
   - **遇到第一条 `xid >= xmin`（其事务可能仍 in-flight）→ 立即停止本轮**，不处理它、不处理其后任何行（即使后面有 `xid < xmin` 的也不抢跑）。
4. **只把已处理的连续安全前缀的末尾 seq 提交为新 cursor**（cursor 与处理副作用同事务，§3.3）。下一轮 `xmin` 推进后，从该 cursor 继续，自然带上那条曾经 unsafe、现已提交的行。

这样 cursor 永远 `< 任何 in-flight 事务最早写入行的 seq`，**永不跨 in-flight 事务、永不漏读低 seq 晚提交的事件**。

**权威 SQL（取批，注意无 xid 过滤）**：

```sql
-- 70 唯一权威取批：不在 SQL 过滤 xid，水位判定交应用层连续前缀扫描
SELECT seq, event_id, topic, payload, xid
FROM outbox_events
WHERE topic = :topic
  AND seq > :cursorSeq
ORDER BY seq ASC
LIMIT :batch;
-- 应用层：取 xmin = pg_snapshot_xmin(pg_current_snapshot())；顺序扫描，
--   命中首条 xid >= xmin 立即 break；只提交 break 之前连续处理过的最大 seq 为新 cursor。
```

**约束声明**：70-events §3.2 第 2 步「`WHERE ... AND xid < :xmin ...`」**作废**，必须改为本节「SQL 不过滤 xid、应用层连续安全前缀停在首条 unsafe」算法。这是所有顺序消费者（MarketplaceProjection / NotifyConsumer / 未来 MeteringConsumer）引用的**唯一权威水位算法**，不得各自变体。

### 11.E 血缘约束注册表（Codex#2 / #6 · 复合 FK 固定约束名）

**问题**：candidate / segment / evidence 三表此前只各自 FK 到对方主键，DB 允许「candidate 属 snapshot A、segment 属 snapshot B、evidence.snapshot_id 伪填」；`current_version_id` / `version_id` 只 FK「任意 version」，不保证版本属同一 capability。需要**复合唯一键 + 复合 FK** 在 schema 层焊死血缘。

**裁定**：以下约束名与模式为**唯一权威**，30 / 40 / 50 域照此命名、照此实现，不得改名或弱化为单列 FK。

| 约束名（固定） | 表 | 模式 | 作用 | 引用域 |
|---|---|---|---|---|
| `uq_session_segments_id_snapshot` | `session_segments` | `UNIQUE (id, snapshot_id)` | 为复合 FK 提供被引用唯一键 | 20 建表、30 引用 |
| `uq_candidates_id_snapshot` | `capability_candidates` | `UNIQUE (id, snapshot_id)` | 同上（候选侧） | 30 建表、30 引用 |
| `fk_evidence_candidate_snapshot` | `candidate_evidence` | `FOREIGN KEY (candidate_id, snapshot_id) REFERENCES capability_candidates (id, snapshot_id)` | evidence 的候选与快照必须配套 | 30 |
| `fk_evidence_segment_snapshot` | `candidate_evidence` | `FOREIGN KEY (segment_id, snapshot_id) REFERENCES session_segments (id, snapshot_id)` | evidence 的段与快照必须配套 | 30 |
| `uq_capability_versions_capability_id` | `capability_versions` | `UNIQUE (capability_id, id)` | 为「版本属同一 capability」复合 FK 提供被引用键 | 40 建表、50 / 70 引用 |
| `fk_publications_capability_version` | `publications` | `FOREIGN KEY (capability_id, current_version_id) REFERENCES capability_versions (capability_id, id)` | current_version 必属同一 capability | 50 |
| `fk_listings_capability_version` | `marketplace_listings` | `FOREIGN KEY (capability_id, version_id) REFERENCES capability_versions (capability_id, id)` | listing 版本必属同一 capability | 50 |
| `fk_runtime_sessions_capability_version` | `runtime_sessions` | `FOREIGN KEY (capability_id, version_id) REFERENCES capability_versions (capability_id, id)` | session 的 version 必属同一 capability（替原两条单列 FK，Codex#6-r2） | 70 |

> 说明：复合 FK 的前提是被引用表上有对应**复合唯一键**。故 30 须给 `capability_candidates` 加 `(id, snapshot_id)` 唯一、`session_segments`（或其可寻址等价表）加 `(id, snapshot_id)` 唯一；40 须给 `capability_versions` 加 `(capability_id, id)` 唯一（与现存 `(capability_id, version)` 唯一并存，前者供 FK、后者防重号）。`candidate_evidence` 的 `snapshot_id` 由两条复合 FK 共同约束，DB 层杜绝「证据跨快照 / 候选段不同源」。`publications` / `marketplace_listings` / `runtime_sessions` 须带 `capability_id` 列（已有）参与复合 FK，保证「current/对外版本/会话版本属于同一 capability」。`capability_versions` 的 `(capability_id, id)` 唯一键由 40 一处建、供 50（publications/listings）与 70（runtime_sessions）共同引用。

**约束声明**：30 的 `candidate_evidence` 现存单列 FK（`candidate_id→candidate(id)`、`segment_id→segment(id)`、`snapshot_id→snapshot(id)`）**升级**为上表两条复合 FK（单列 snapshot FK 可保留为冗余完整性，但血缘正确性由复合 FK 保证）；50 的 `publications.current_version_id` / `marketplace_listings.version_id` 单列 FK **升级**为上表复合 FK；70 的 `runtime_sessions` 原两条单列 FK（`capability_id→capabilities`、`version_id→capability_versions(id)`）**替换**为单条复合 FK `fk_runtime_sessions_capability_version (capability_id, version_id) → capability_versions(capability_id, id)`（Codex#6-r2，删除原单列 FK）。约束名不得偏离。

### 11.F 社交写权限：任意已登录用户（Codex#17）

**问题**：10-auth §6.2 把「社交写」挂 `requireRole('creator')`，60-profile 说「任意登录用户可 follow/like」，两域口径冲突。

**裁定（唯一权威，10 / 60 对齐）**：**follow / like 等社交写接口对「任意已登录用户（authenticated user）」开放，不限 creator role。** 口径理由：社交关注/点赞是消费侧基础互动，限定 creator 会让普通登录用户无法关注创作者，违背产品意图。

**权威权限口径**：
- 社交写（`POST /follows`、`POST /likes`、及其 DELETE 取消）守卫 = **`requireAuth`**（任意已登录用户），**不**加 `requireRole('creator')`。
- **POST 与 DELETE 都是写命令，统一遵 §4「所有写操作必带 `Idempotency-Key`」**（不因 DELETE 天然幂等而豁免），各带**固定 scope**：`POST /follows`=`social.follow`、`DELETE /follows`=`social.unfollow`、`POST /likes`=`social.like`、`DELETE /likes`=`social.unlike`。重复请求按 §4 行为矩阵回放首次结果、不重复增减计数。仍需登录（匿名 `optionalAuth` 不可写社交，未登录写 → `401 UNAUTHENTICATED` + `action:'escalate'`）。
- 自己不能关注/点赞自己等业务校验由 60 域定义，与权限口径无关。

**约束声明**：10-auth §6.2 表中「社交写 → `requireRole('creator')`」**修正为** `requireAuth`；60-profile 按此挂载，两域权限表、错误用例、前端权限态一致。引用方：10（权限表）、60（社交写端点）。

### 11.G 后置 FK 闭合清单（Codex#18-r4 · drafts 建表顺序环破除，全 38 表 FK 可建得出、无环）

**问题**：`drafts`（核心表，紧随 `users`/`jobs` 建）的落点引用列指向后建的导入/结构化/发布域表（`raw_snapshots`/`capability_versions`/`publish_batches`），而导入域 `import_pairings.draft_id` 又反向引用 `drafts`——内联这些 FK 会形成「建表顺序环」；且 `drafts.batch_id` 此前缺 `publish_batches(id)` FK。

**裁定（唯一权威）**：跨域/前向引用的 FK 一律 **「先建基表 + 后置 `ALTER ADD CONSTRAINT`」**（与既有 40 域 `fk_capabilities_current_version`、70 §9.4 冻结表后置 FK 同模式）。全契约后置 ALTER FK 闭合清单如下，按依赖**统一在所有基表建好后执行**，约束名固定、供迁移与合并校验：
