# Agora 创作者中心 · 主链路技术方案（架构 / 链路 / 选型 / 部署）

> 覆盖创作者中心的主流程：导入、提取、选择、结构化、发布五步。试用（Trial）还在设计中，本文不展开。
>
> 对应 PRD：飞书《Agora · 创作者中心 PRD》。现状代码：仓库 `agora-mvp`。

---

## 〇、现状与目标

### 0.1 现在是什么样

现在是一个最小可用版本：一个 Node 服务文件扛起全部接口，数据存在一个 JSON 文件里，部署在 Railway 上。

它已经跑通了主流程的核心算法：会话解析、能力聚类、能力说明书（manifest）生成、进度推送。这些是可以直接复用的资产。

但它没有数据库、没有任务队列、没有多进程，耗时的 AI 处理和接口响应挤在一个进程里。这一版要做的，就是把它产品化。

### 0.2 两处设计考虑

**流程预演。** 这一版里，本机只把对话原文整份上传。云端负责后面所有整理：解析、去敏、提取、结构化、发布。去敏（抹掉手机号、密钥这类隐私信息）是这一版新增的步骤。

**持久化与多进程化。** 数据从单个 JSON 文件换成正式数据库。服务按职责拆成三个进程：一个接请求、一个跑后台任务、一个收事件，各干各的、互不拖累。

### 0.3 目标

- 五步流程可以随时存草稿、断点续传；耗时步骤不转圈，而是把进度拆成一条条子任务实时点亮，结果生成一段就先显示一段。
- 分层清晰、不绕圈依赖；进程之间只通过数据库、队列、事件通信。
- 关键数据有事务、有唯一真源；发布是一次完整的事务。
- 性能要求不高，优先用 Docker Compose 单机部署。
- 给消费端（市集/社区）预留好数据与接口，消费端界面没定也不影响主流程。

### 1.1 模块架构（图1）

【此处嵌入图1 · 模块架构图】

### 1.2 整体说明

系统是前后端分离的 Web 应用，后端是一个模块化单体，按三类进程跑：接请求的 API、跑后台任务的 Worker、收事件的 Consumer。三者共用同一套内核与数据访问，但进程隔离，AI 萃取不会卡住接口。

| 模块 | 它负责什么 |
|---|---|
| 前端应用 | 承载导航外壳和五步上传向导，实时展示处理进度，支持草稿续传 |
| 本机导入助手 | 装在创作者电脑上，扫描本机对话历史并把原文整份上传 |
| API 服务 | 对外的唯一入口，负责鉴权、收上传、建任务、查询、推送进度 |
| 后台 Worker | 跑四类耗时任务：导入解析、能力提取、能力结构化、发布 |
| 平台内核 | 沉淀可复用的能力：会话解析、去敏、能力说明书生成、任务与事件、AI 调用网关 |
| 数据层 | 数据库存业务真源，Redis 做队列与缓存，对象存储放快照、封面和产物 |
| 消费链路 | 接收发布事件，把能力被调用的次数和消耗写回创作者工作台（界面待定，先把接口定下来）|

### 1.3 分层规则

依赖只能从上往下：前端到 API，API 和 Worker 和 Consumer 都向下用内核、再到数据层，三类进程之间互不直接调用。这样不会出现绕圈依赖。各层的连接关系见图1，箭头表示调用方向。

## 二、主链路详细设计

### 2.1 主链路时序（图2）

一段本机对话历史，怎么一步步变成市集上的能力，整条链路见下图。

【此处嵌入图2 · 主链路时序图】

下面按五步说明每一步做什么、产生什么数据。

### 2.2 第一步：导入

创作者用本机助手把对话原文整份传到云端。Worker 把原文解析成一段段会话，去敏后存成一份会话快照；原文用完即删，正式存储里只留去敏后的内容。重新导入会生成新的一份快照，旧的保留。

数据结构：

```sql
raw_snapshots(id, user_id, s3_key, segments_cnt, messages_cnt, span, redaction_report jsonb, created_at)
session_segments(id, snapshot_id, user_id, title, source, project, content_ref,
                 content_hash, msg_cnt, happened_at,
                 UNIQUE(snapshot_id, content_hash))   -- 同一份快照内按去敏后内容去重，重复落库不报错
```

进度用任务（jobs）记录，前端订阅后实时显示：子任务一条条点亮，可后台执行、可取消。

### 2.3 第二步：提取

AI 扫描会话段，识别出反复出现、值得打包成能力的工作流，按频率和可打包度排序，识别出一个就先显示一个、供创作者勾选。单个识别失败只影响那一行，可单独重试，不挡其他。

数据结构：

```sql
capability_candidates(id, extract_job_id, snapshot_id, name, slug, tagline, type,
                      confidence, freq_segments, order_index, candidate_hash, llm_run_id,
                      status, error jsonb, retry_cnt, created_at, updated_at,
                      UNIQUE(extract_job_id, slug))
candidate_evidence(candidate_id, segment_id, quote_excerpt, PK(candidate_id, segment_id))
                      -- 记录每个能力由哪些会话段支撑（段级血缘）
```

### 2.4 第三步：选择

创作者挑一个能力进入下一步，或者直接选「全部发布」。这一步纯前端操作，不调模型、不写库、不等待。

选择结果跟着下一步的请求一起发给后端，请求里带一个去重标记，重复点或刷新页面也只会发布一次、不会重复跑。选「全部发布」时，后端会对这批能力逐个跑结构化和发布。

### 2.5 第四步：结构化

把选中的能力整理成一份能力说明书。它分两部分：软字段由 AI 生成、创作者可改可重生成；硬字段由平台锁定。某个字段生成卡住时，可以用已生成的部分先继续、或只重生成卡住的那个，已生成的内容不丢。

落库时说明书沿用现有的扁平结构（manifest v0.1），「软/硬」只是界面上的分组和锁定标记。各字段对应到说明书里的位置如下表。

| 说明书字段 | 软/硬 | 落库到说明书的哪个字段 |
|---|---|---|
| 名称、卖点、角色、目标、步骤、技能、起手示例 | 软 | 说明书对应字段（manifest.name / agent.role 等）|
| 唯一标识、版本、状态 | 硬 | 由平台填写并锁定 |
| 输入项、产物形态、风险红线 | 硬 | 字段名由系统从指令里抽出，类型和是否必填由平台补全 |

数据结构：

```sql
capabilities(id, creator_id, slug, name, audience_tags[], domain_tags[], scene_tags[],
             current_version_id, total_invocations, freq_segments,
             UNIQUE(creator_id, slug))           -- 标识按创作者维度唯一，创建后不变
capability_versions(id, capability_id, semver, manifest jsonb, manifest_hash, status,
                    structure_state jsonb, from_candidate_id, created_at)
```

### 2.6 第五步：发布

发布前，创作者先看一张「市集卡预览」，就是消费者将来在市集里看到的样子：封面、名称、卖点、价格、试用按钮。名称和卖点来自上一步的软字段，封面和价格创作者自己定。

点发布后，后端在一次事务里完成全部动作：冻结这一版说明书、写发布记录、固化价格、生成市集卡、记一条上架事件。能力随即上架，但带「Alpha · 审核中」徽章，进入人工评审。

评审通过就清掉徽章；被拒绝就退回上一版，没有上一版就下架。整个状态流转见下图。

【此处嵌入图3 · 发布状态流转】

数据结构：

```sql
publications(id, capability_id UNIQUE, version_id, share_token UNIQUE, url_slug UNIQUE,
             visibility, review_status, published_at)
capability_tiers(id, version_id, tier_code, name, price_micros, quota jsonb,
                 UNIQUE(version_id, tier_code))      -- 价格在发布时固化，是定价唯一真源
marketplace_listings(capability_id PK, version_id, manifest_hash, card jsonb,
                     search_tsv tsvector, status, updated_at)
outbox_events(id BIGSERIAL, event_id uuid UNIQUE, topic, aggregate_id, payload jsonb,
              processed_at, created_at)
```

发布事件和业务数据写在同一个事务里，保证不丢不重。消费端单进程按「没处理过的先处理、处理完打个标记」的方式消费，配上事件去重，不会因为编号乱序漏掉事件。

留给消费端的接口已冻结：读市集目录、读单个能力、读某版说明书；运行结束后把用量写回工作台。这些接口的存在不依赖消费端界面定稿。

### 2.7 三个贯穿机制

**任务。** 导入、提取、结构化、发布都登记成一条任务记录，状态以数据库为唯一真源，队列只负责触发。每条任务有重复保护，执行时占一个有时限的执行权；超时后由一个专门的巡查进程接手，能安全重跑的就重跑、不能的就标记失败等人工。

**进度。** 前端订阅任务进度，子任务一条条点亮、结果生成一段就先显示一段，断线能接着传。

**草稿。** 每步可存草稿，工作台的草稿条记录停在哪一步，一键回到中断处。

异常统一三条：不停在转圈、不甩错误码（都落成带退路的错误态）、已生成的内容不丢。

## 三、技术选型

### 3.1 选型一览

| 用在哪 | 现在用什么 | 换成什么 | 为什么 |
|---|---|---|---|
| 后端框架 | 原生 http 手写路由 | Fastify | 内建请求校验和日志，性能好，便于分层 |
| 语言 | JavaScript | TypeScript | 数据结构复杂，类型能保证各层契约对得上 |
| 数据库 | 一个 JSON 文件 | PostgreSQL | 要事务、并发、聚合统计、全文搜索、存说明书 |
| 队列/缓存 | 无 | Redis + BullMQ | 耗时任务异步化、进度续传、跨副本共享、限流 |
| 对象存储 | 本地文件 | MinIO（兼容 S3）| 放暂存原文、去敏快照、封面和产物，自托管省钱 |
| AI 调用 | 散在各处 | 统一的 AI 网关 | 集中做限流、重试、计费、流式 |
| 事件 | 无 | 数据库事务型事件表 | 同库低量级，和业务写同事务最省心，不必上 Kafka |
| 前端 | 静态 HTML | React + Vite | 五步向导有状态机、要断点续传和进度刷新 |

### 3.2 几点说明

去敏是这一版新增的能力，现在代码里没有，要单独设计规则和测试，是导入环节里最需要谨慎的一块。

能力说明书继续沿用现有的扁平结构，不另起新格式，这样运行端和消费端都不用改。

事件用数据库事务型事件表就够了。真正需要换 Kafka 的情形是「将来有多个跨团队的消费方」，按现在的量级短期不会发生。

向量检索（用来做市集里的相似推荐）属于消费端，不在主流程，等做消费端时再加。

### 3.3 迁移顺序

按可验证的小步走，不停机重写：先抽出内核、再起数据库迁数据、再拆三个进程、再换前端、最后容器化。

有一个硬性顺序：必须先把数据迁到数据库，再拆多进程。因为现在的数据是存在单进程内存里的，一旦拆成多进程，进程之间读不到对方的内存。

## 四、部署与运维

### 4.1 部署拓扑（图4）

一台云服务器，用 Docker Compose 把全栈拉起来即可（4 核 8G 起步）。容器构成和数据持久化见下图。

【此处嵌入图4 · 部署拓扑图】

| 容器 | 干什么 | 数据存哪 |
|---|---|---|
| web（nginx）| 托管前端、反向代理接口、透传流式连接 | — |
| api | 接请求、建任务、推进度 | — |
| worker | 跑导入、提取、结构化、发布任务 | — |
| sweeper | 巡查超时或卡住的任务并接手处理，全局只跑一个、不随 worker 扩缩 | — |
| consumer | 接收事件、把用量写回工作台 | — |
| postgres | 业务真源 | 命名卷 pgdata |
| redis | 队列、进度续传、缓存 | 命名卷 redisdata |
| minio | 暂存原文桶、正式快照桶 | 命名卷 miniodata |

api、worker、sweeper、consumer 是同一个镜像、不同启动入口，按负载各自扩缩。数据库、Redis、MinIO 都配健康检查，应用容器等它们就绪了才启动。nginx 代理流式连接要专门配置（用 HTTP/1.1、关缓冲、加长读超时），否则进度推送会断。

### 4.2 部署流程

先起依赖、再迁数据库、迁成功了才起业务容器，避免新代码碰到旧表结构。

```bash
# 1. CI 产出不可变镜像（按 git 提交号打标签，生产机不现场构建）
docker build -t registry/agora:<git-sha> .
docker build -t registry/agora-web:<git-sha> ./web

# 2. 先起依赖，再迁移（迁移失败即中止，不起业务容器）
docker compose up -d postgres redis minio
docker compose run --rm migrate npm run migrate

# 3. 迁移成功后再起/更新业务容器
docker compose up -d --no-deps api worker sweeper consumer web

# 4. 健康检查
curl -f http://localhost/api/v1/ready
```

更新和回滚都靠不可变镜像：更新就拉新标签重启，回滚就指回上一个提交号。数据库迁移只加不减、向后兼容，保证回滚安全。

### 4.3 运维

| 方面 | 怎么做 |
|---|---|
| 日志 | 结构化 JSON 输出，按任务号串联，采集到云日志 |
| 监控 | 健康检查只看数据库、Redis、MinIO；外部 AI 服务不稳定时只标记为状态变差，不影响整套系统的健康判定、也不停掉对外服务 |
| 告警 | 队列积压、任务失败率、事件卡住、用量没记上，都触发告警 |
| 备份 | 数据库每日备份到异地（不放在自己的 MinIO 里），定期演练恢复，明确恢复目标 |
| 扩缩容 | worker、consumer 无状态可水平扩；巡查进程固定一个 |
| 数据安全 | 原文只进加密暂存桶、用完即删；正式桶只存去敏后内容，私有桶加签名 URL |
| 密钥 | 用 Docker secrets 挂载，不进镜像、不进代码库；定期轮换 |

### 4.4 CI/CD

代码推上来后，流水线依次跑：检查（含分层规则）、测试（数据库、Redis、MinIO 用临时容器）、构建镜像、推仓库、部署、上线后跑一遍最基本的可用性检查。

环境分三套：本地开发、预发（合并后自动部署、再跑一遍可用性检查）、生产（打标签或人工审批）。检查、测试全过才允许构建；违反分层规则就判定流水线失败、不许合并。
