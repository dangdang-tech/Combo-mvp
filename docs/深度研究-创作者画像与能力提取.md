# 深度研究 · 从 AI 对话历史萃取「创作者画像 + 能力」:方法 / 结构体 / 可视化

> 三路并行深挖真实开源项目源码、官方文档、论文(2023–2026)。每个结论尽量给真实字段名、真实 prompt 思路、真实算法与源链接;查不到的明确标注。这是 Agora「raw → 能力 + Personal Context」一层的研究底座。

## 0. 一句话定位

**本产品本质 = Clio(无监督归纳"人们用 AI 做什么")+ Voyager 的 skill 结构 + Generative Agents 的 reflection(支持度阈值触发),跑在【单个用户】的会话日志上,而非百万用户上。**
- 画像层(Personal Context)→ 借 **Honcho** 的"explicit→inductive 推理"+ **Letta** 的 human block + **Zep** 的双时态演化。
- 能力层(capabilities)→ 借 **Clio** 的层级聚类+对比命名、**Voyager/Trace2Skill** 的 skill 结构、**频繁序列挖掘**做客观频次锚。
- 关系层(图谱)→ 借 **mem0ᵍ/Zep** 的实体-关系图。

## 1. 用户记忆 / 画像系统:真实结构与提取方法

### 1.1 mem0 — 两阶段被动抽取(extract → update)
- **提取**:先 fact extraction(7 类:偏好/个人细节/计划/活动/健康/职业/杂项,只从 user+assistant、不从 system,且按输入语言记录),再 memory update 做 **ADD/UPDATE/DELETE/NONE** 决策(保 ID、矛盾则 DELETE、语义等价则 NOOP)。
- **结构**:扁平 fact(向量行);图变体 mem0ᵍ 用三元组 `(source, relationship, destination)` + 工具 `add/update/delete_graph_memory`(更新只改 relationship、节点不变;删除有"同关系不同 destination 不删"的启发式)。
- **演化**:原地改写保 ID + history;图侧"更准覆盖 + 矛盾删边"(非时序)。
- 源:[prompts.py](https://github.com/mem0ai/mem0/blob/main/mem0/configs/prompts.py) · graph [tools.py@v0.1.114](https://github.com/mem0ai/mem0/blob/v0.1.114/mem0/graphs/tools.py) · [arXiv:2504.19413](https://arxiv.org/abs/2504.19413)

### 1.2 Letta / MemGPT — agent 自编辑文本块
- **结构**:memory **block**(`value` 文本 + `limit`≈2000字符 + `label`(human/persona)+ `description` + `read_only`)。三层:core(每轮进 context)/ recall(全历史可搜)/ archival(外部向量库)。
- **方法**:agent 调工具自己改 —— `core_memory_append/replace`、新版 line-based `memory_insert/replace/rethink`(整块重写压缩)。context 逼近上限触发"memory pressure"搬运。`human` block = 一段不断演化的用户叙述。
- 源:[block.py](https://github.com/letta-ai/letta/blob/main/letta/schemas/block.py) · [base.py 工具](https://github.com/letta-ai/letta/blob/main/letta/functions/function_sets/base.py) · [MemGPT arXiv:2310.08560](https://arxiv.org/abs/2310.08560)

### 1.3 Honcho — 形式逻辑做 Theory-of-Mind 画像(最贴"画像")
- **结构**:对每个 peer 提 explicit atomic facts,再做 **deductive/inductive/abductive** 推理。输出 `{ "explicit":[{content}], "deductive":[{premises:[], conclusion}] }`;representation = 结论 + summaries(每 20/60 条消息)+ **peer cards**(name/occupation/interests 缓存)。按 **(observer, observed)** 非对称分片("Alice 眼里的 Bob")。带 **reasoning tree**(premises↔conclusions)。
- **方法**:后台 deriver(攒 ~1000 tokens 才跑一次)；抽取规则:转成 self-contained 结论、用绝对日期、正确归属主体、常识桥接。查询走 agentic Dialectic(强制先查 preferences)。
- **演化**:每条消息 refine + **consolidation 显式消解矛盾**;空闲期 Dream Specialist 巩固。
- 源:[deriver/prompts.py](https://github.com/plastic-labs/honcho/blob/main/src/deriver/prompts.py) · [reasoning.mdx](https://github.com/plastic-labs/honcho/blob/main/docs/v3/documentation/core-concepts/reasoning.mdx)

### 1.4 Zep / Graphiti — 双时态知识图(演化最强)
- **结构**:EpisodicNode(原始事件)/ EntityNode(`name_embedding`+`summary`+`attributes`)/ EntityEdge(`fact`+`fact_embedding`+`valid_at/invalid_at/expired_at`)。**双时态四时间戳**:t_created/t_expired(事务)+ t_valid/t_invalid(现实)。
- **方法**:`add_episode` = 抽实体 → 抽带时间的边 → entity resolution/dedup。边抽取:`relation_type` 派生 **SCREAMING_SNAKE_CASE**、ISO8601 时间、**保留所有专有名词/型号/数量(加细节算新 fact 不算重复)**。
- **演化**:矛盾**不删**,旧边标 expired/invalid 新建一条 → 完整历史,可查任意时点。
- 源:[nodes.py](https://github.com/getzep/graphiti/blob/main/graphiti_core/nodes.py) · [extract_edges.py](https://github.com/getzep/graphiti/blob/main/graphiti_core/prompts/extract_edges.py) · [Zep arXiv:2501.13956](https://arxiv.org/abs/2501.13956)

### 1.5 Stanford Generative Agents — memory stream + reflection
- **结构**:每条 memory = 自然语言 `description` + 创建/最近访问时间戳 + importance 分;类型 observation/reflection/plan;reflection 节点带"指向证据 memory 的指针"。
- **方法**:检索 `score = recency + importance + relevance`(α 全 1、min-max 归一);recency 指数衰减 **0.995**;importance LLM 1–10 打分(有原文 prompt)。**reflection 触发 = 最近事件 importance 累加超 150**(约每天 2-3 次)→ 提 3 个高层问题 → 推 5 条洞察(带 because-of 证据编号)→ 存回形成洞察树。
- 源:[arXiv:2304.03442](https://ar5iv.labs.arxiv.org/html/2304.03442)

### 横向对比
| 维度 | mem0 | Letta | Honcho | Zep/Graphiti | Gen-Agents |
|---|---|---|---|---|---|
| 提取 | 被动 fact+AUD决策 | agent 自编辑 | 被动 deriver+逻辑推理 | 被动 实体+时序边 | append+周期 reflection |
| 结构 | 扁平 fact / 三元组 | 文本 block | 推理树+peer card | 双时态图 | 记忆流 |
| 演化 | 改写保ID | append/rethink | refine+consolidate | 不删标失效(最强) | 衰减+反思上卷 |
| 矛盾 | DELETE | agent replace | consolidation | 双时态失效 | 无 |

## 2. 能力 / 反复任务 的提取:真实方法

### 2.1 关键来源
- **Clio**(本产品直接蓝图,arXiv:2412.13678):从海量对话无监督归纳"人们用 AI 做什么"的层级 taxonomy。四步:① facet 抽取(每对话抽主题等,省隐私)② facet embedding → k-means base 聚类 ③ **对比式命名**(给 LLM 簇内样本 + 最近邻簇负例,生成既概括又区分邻簇的名 → 防命名漂移)④ 自底向上建层级(neighborhood≈40个簇、提议父类、跨域去重、分配、re-summarize、递归)。**支持度阈值**:无 cluster 少于 ~1000 用户 → 单用户场景退化成"能力至少 N 段会话支持才展示"。
- **Voyager**(arXiv:2305.16291):只把**自检通过/成功**的轨迹固化成 skill;skill = {语义化函数名, 代码, 自然语言描述, **描述的 embedding**};**按描述 embedding 检索**(意图层对齐),top-5 复用。→ 启示:只从成功会话抽能力;检索/去重走"意图描述"而非原文。
- **Generative Agents reflection**:给了能力的 **evidence 指针**怎么填 + importance 怎么打 + 何时触发归纳(累计权重阈值,频繁/重要更快被提炼)。
- **Trace2Skill**(arXiv:2603.25158):skill = `{ preconditions, action_sequence, postconditions, generalization_patterns }`;流程:切分轨迹 → 抽决策点 → pattern abstraction(具体→泛化)→ 相似聚类合并等价 skill。**schema 几乎可直接采用。**
- **SkillGenBench**(arXiv:2605.18693):skill 标准化成 `SKILL.md`(文档+步骤+**接口契约 inputs/outputs**+grounding);关键发现"结构对但执行错"→ **evidence/grounding(指回真实会话)必须硬性带上**。
- **task/process mining**:把会话 → 归一动作 token 序列;**频繁序列挖掘(PrefixSpan/SPADE)** 算 support_count → 给"频次"一个**不依赖 LLM 的客观锚**;先 trace clustering 再挖,避免串味。
- **稳定性**:LLM 抽取不稳 → 同输入采样 3-5 次,对"抽出的集合"按元素多数投票,**Jaccard** 度量稳定性 → confidence = 出现率 × Jaccard。

### 2.2 推荐的「能力」结构体(综合上述)
```jsonc
{
  "capability_id": "uuid",
  "name": "调试 React hydration 报错",        // LLM 对比式命名(带邻簇负例),稳定后锁定
  "canonical_slug": "debug-react-hydration",  // 稳定标识,改名不变(跨运行 diff)
  "intent_summary": "...",                    // 检索 key 的来源
  "action_steps": ["..."],                    // process discovery / LLM 归纳
  "preconditions": ["..."], "inputs": [...], "artifacts": [...],  // Trace2Skill + 接口契约
  "tools_used": ["Bash","Edit","gh"],         // 客观可统计
  "evidence": [{"session_id","message_span","snippet","outcome"}], // 硬性:指回真实会话
  "frequency": { "support_count":17, "occurrences":41, "first_seen","last_seen", "recency_weighted" },
  "importance": 0.0,   // norm(poignancy 1-10) × log(1+support) × artifact_value
  "confidence": 0.0,   // 出现率 × Jaccard 稳定性
  "project": ["repo-A"], "cluster_path": ["代码维护","调试","前端调试"],
  "status": "active|stale|merged",            // Zep 式:旧能力标失效不删
  "related": [...],                            // 见 2.4
  "extraction_meta": {"n_samples":5,"appeared_in":4,"jaccard_stability":0.78}
}
```
填法速记:name/steps/pre/artifacts 用 LLM(只成功片段);support_count/tools_used **非 LLM**(频繁挖掘+统计,做客观锚);confidence 用多采样+Jaccard;evidence 必须可 drill-down 回会话。

### 2.3 推荐的提取管线(三层:客观频次 → LLM 抽取共识 → 聚类命名)
0. **归一化**:解析 JSONL,每会话 → (意图, 工具调用序列, 产物),推断 project(cwd/git remote)。
1. **客观频繁模式(无 LLM)**:trace clustering → PrefixSpan 挖 support≥θ 的模式 + support_count → 频次硬锚,不会漂。
2. **LLM 抽候选(多采样共识)**:mem0 式 prompt(整体摘要+局部滑窗+当前片段),每段独立采样 3-5 次,按元素多数投票 + Jaccard,`confidence=出现率×稳定性`,低于阈值丢弃。
3. **去重/合并(mem0 update 算子)**:候选 intent embedding 检索 top-10 已有能力,LLM 走 ADD/UPDATE(support+=、合 evidence)/DELETE(标 stale 不删)/NOOP → 增量稳定,不长近义。
4. **支持度阈值过滤(Clio)**:只留 support_count≥N(如≥3 段);频次锚(1)与 LLM(2)**交叉**:都指向=高可信,仅一边=tentative。
5. **聚类+层级+对比命名(Clio 算法)**:intent embedding → base 聚类 → 对比命名(带负例)→ 自底向上建层级 → cluster_path。**命名稳定后写 canonical_slug,后续复用 taxonomy 分类(TnT-LLM 式)而非重新归纳**(防漂)。
6. **打分+周期触发(Gen-Agents)**:算 importance/recency;**累计权重超阈值才触发**全量 reflection/重聚类。

> 与现有实现的关系:我们当前的 v3 锚定管线(S1 精读 → 建 taxonomy 一次 → K 轮分类 + Jaccard)已经实现了【2 共识 + 4 阈值 + 5 命名稳定】的核心。**待补**:第 1 步的客观频繁锚(给频次去 LLM 化、更稳)、evidence 指针硬性化、status 失效(双时态演化)、能力间关系边(2.4)。

### 2.4 能力间关系(可计算定义,每边权重 ∈[0,1])
1. `SIMILAR_TO` = `cosine(intentA, intentB)`,≥0.8 保留(近重去重/合并候选)。
2. `CO_PROJECT` = `Jaccard(projectA, projectB)` —— **这就是我们图谱现在用的"共享项目"边**。
3. `CO_OCCUR` = 会话级 lift `P(A,B)/(P(A)P(B))`,`w=sigmoid(log lift)`,>1 = 工作流式关联。
4. `PRECEDES` = 频繁序列里 A 常紧接 B 之前,`w=count(A→B)/count(A)` → 工作流 DAG。
5. `PARENT_OF` = Clio cluster_path(确定性)。
6. `GENERALIZES` = A.steps⊇B.steps 且前提更宽。
> 现状:我们只用了 `CO_PROJECT`。**升级方向**:加 `CO_OCCUR`(同会话共现)和 `PRECEDES`(工作流顺序)才是真"网状"——现在 agora 簇之外都是孤点,正因只有一种边。

## 3. 可视化:真实先例 + 落地规范

### 3.1 真实产品先例(画像)
- **LinkedIn Skills**:只置顶 Top 3、其余折叠;每技能带背书数+头像(社会证明)。→ **技能不平等罗列,要有主次、可策展、带证据(数量)。**
- **GitHub 贡献热力图**:`{date,count}` → calendar heatmap,密度叙事。→ 时间维度的投入,热力图最稳。
- **Spotify Wrapped(最强参考)**:**没有一张图表**,靠大字号数字(BAN)+ 渐变 + **4 字母"性格类型"标签** + 滑卡叙事。原则:只挑几个高光、文字吝啬色彩慷慨、数据情感化。→ **画像 = 一个"创作者类型"标签 + 3-5 个 BAN 核心数字**,不是 dashboard。
- **Strava**:近4周/年度/全部 **三时间尺度并置** → 既看势头又看积累。
- **Obsidian graph**:力导向"看着爽、actionable 弱"(易 hairball)→ **图负责氛围/发现,actionable 信息靠点击进 side panel。**

### 3.2 知识图谱可视化最佳实践
- 节点:大小=重要性/度数(视觉地标)、颜色=类别/社区(Louvain,最多 6-8 簇色)、形状≤2-3 种、标签用可读名仅大节点常显。
- 边:粗细=强度、颜色=关系类型(克制)、方向有语义才显箭头。
- 布局:**Radial / ego-centric(用户中心、能力放射)比纯力导向更可控、不易 hairball** —— 这正是我们选的。
- 防 hairball:LOD(先 50-100 节点)、渐进披露(expand-on-demand)、过滤、聚类折叠、Focus+Context(点节点高亮 2 跳邻域其余淡出)、节点上限+"+N"展开。
- 库:react-force-graph(Canvas)起步;样式需求高换 Cytoscape.js;破千节点上 Sigma.js(WebGL)。**6 节点静态稿:确定性极坐标自渲染 SVG 最不丑、最可复刻**(我们已采用)。
- 源:[yFiles 知识图谱指南](https://www.yfiles.com/resources/how-to/guide-to-visualizing-knowledge-graphs) · [Grafana node graph](https://grafana.com/docs/grafana/latest/visualizations/panels-visualizations/visualizations/node-graph/)

### 3.3 Personal Context 画像面板信息架构(推荐)
遵循 Summary→Diagnostic→Detail 三层 + 5 秒规则(3-5 个核心指标先读到、一屏 5-9 元素、深度信息渐进披露):
- **区A 身份 Hero**:头像/名 + **"创作者类型"标签**(萃取的一句话原型,如"探索型系统构建者",学 Spotify)+ 一句定位。大字、留白、一主色。
- **区B 核心数字带(3-5 个 BAN)**:对话总量 / 能力点数 / 知识域数 / 最活跃主题 / 成长最快能力;可加近30天/全部双尺度(学 Strava)。
- **区C 能力分布(Diagnostic)**:Top 5-8 能力,带 **数据条**(强度)+ 证据数(类比背书数)+ 趋势;**置顶 Top 3 其余折叠**(学 LinkedIn);每条可点 → side panel 出证据会话。
- **区D 活跃度热力**:GitHub 式 calendar heatmap。
- **区E 知识域概览**:3-6 个簇(与图谱同色),点击跳图谱聚焦该簇。

> 对照我们现在的 v5 PC 卡:已有"你最常做的(数据条)/工作风格/活动足迹/可打包度"四块,**已对上区B/C 的精神**。**可补**:① 区A 的"创作者类型"身份标签(现在只有"全栈+AI",可升级成更 telling 的原型);② 区D 活跃度热力(时间维度现在缺);③ 能力条可点 → side panel 出证据(evidence drill-down,信任刚需)。

### 3.4 能力图谱视觉编码 + 移动端
- 编码规范见 `docs/能力关系网络-可视化逻辑-spec.md`(圆大小=√频次、实线=共享项目、虚线=隶属、置信=描边色、按项目 halo 聚簇)。
- **移动端(<640px)**:力导向/放射图在手机上极难点 → 业界共识**降级**:图谱 → **按段数降序的能力列表卡**(名+强度条+域色点+证据数),点某能力 → 进"该能力 ego 小图"(≤15 节点);提供"列表↔图"切换默认列表。PC 卡多列 → 单列堆叠(Hero→BAN→能力排行→热力→域),热力图减周数或转 mini sparkline。

## 4. 对 Agora 的落地结论(synthesis)

1. **Personal Context 最终结构体** = 身份(Letta human block 式一句画像 + Spotify 式"创作者类型"标签)+ 派生特质(Honcho inductive 推理,带证据)+ 能力网络(mem0ᵍ/Zep 实体-关系)+ 足迹(Zep 双时态:可呈现"成长轨迹")+ 可打包度(产品特有 actionable)。**每条画像结论必须可溯源(evidence 指针)** —— 对开发者用户是信任刚需。
2. **能力提取**:我们现有 v3 锚定管线已实现共识+阈值+命名稳定;**优先补**:客观频繁锚(频次去 LLM 化)、evidence 硬性化、status 失效演化、`CO_OCCUR`/`PRECEDES` 关系边(让图真"网")。
3. **演化用双时态不覆盖**(Graphiti):技术栈/偏好会变,旧画像标失效保留 → "成长轨迹"是卖点。
4. **可视化分工**(Obsidian 教训):图负责"全局感/发现",画像卡 + 点击 side panel 负责"有效信息" —— 别指望图本身承载一切。
5. **画像要叙事化**(Spotify):一个"创作者类型"标签 + 少数 BAN + 带证据能力排行,而非堆指标。
