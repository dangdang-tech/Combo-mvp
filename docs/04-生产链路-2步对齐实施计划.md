# 生产链路 · 服务端 2 步对齐实施计划

把 authoring 现有「多步多表」实现，收敛到《生产链路·服务端方案》（Feishu wiki `NRQjwLPyrihVOokX2dwcAMUEnld`）的 2 步目标设计。本文是方案与代码之间的施工说明，供分步落地与评审。

方案目标：两步（导入 → 分析合成）+ 一个 `capabilities` 结构（合并候选/版本/Skill 包/发布侧）；发布是能力上的轻量同步事务；市集目录/价格/批量/复审为非目标。

### H1 — 大文件直传被打碎（需前后端协同，非后端独立可修）

根因在前端：`web` 对 >8MiB 文件按字节切片成多 part（`useBrowserImport.ts`）。后端上传清单（`import_uploads.expected_parts`）每个 part 只有 `clientPartId + s3Key`，**没有「哪些 part 属于同一文件、片内序号」的分组信息**；`rawS3Keys` 是按 `clientPartId` 字典序的全局有序对象列表，worker 直传分支（`import/job.ts:295`）把每个 key 当一个完整会话文件解析。

因此单个 >8MiB 文件被字节切成多 part 后，worker 无元数据判断哪些 part 该拼、按什么序拼，后端无法独立、可靠地重组（启发式拼接 JSONL 边界反而可能制造新的损坏）。

落地方向（需前端改一处）：

- 前端：要么按文件边界（换行）切，单文件不跨 part；要么给每个 part 打 `fileId + partIndex`。
- 后端：`ExpectedPart` 增 `fileId / partIndex`，worker 直传分支按 `fileId` 分组、按 `partIndex` 排序拼接后再解析；分片内容 sha256 硬校验（梳理 P2）一并补。

### M2 — 原文明文留桶（按方案推迟真删）

`import/job.ts:408` 处理完只软标 `raw_purged_at`，注释自称「S3 删对象由 sweeper orphan 驱动」，但 `sweeper-reconcile.ts` 只做租约对账、无任何 S3 删除逻辑；且 `raw_snapshots.raw_s3_key` 只记首片（`job.ts:352` 的 `rawKeys[0]`），`raw_s3_key` 当前全仓无读取方。

方案对 M2 的口径是「标记 + 推迟实删，上线前补孤儿巡查真删」。真删是会删除生产环境明文原文的破坏性新基建，按方案推迟；上线前落地：

- `raw_snapshots` 加 `raw_s3_keys text[]`（存全部 part 键，而非首片），insert 写全部键。
- 新增孤儿巡查：扫 `raw_purged_at IS NOT NULL` 且超保留期的快照，删其全部 S3 键，标记完成。
- 隐私债，呼应《导入二进制方案安全债》，上线前必修。

### P5 — skill_set 文本退役（待 Skill 包落地）

`ManifestSchema` 仍含 `skill_set: string[]`，且作为 7 个软字段之一被 `structure/generate.ts` 生成、落 `capability_versions.manifest`。方案要它退役、由落对象存储的真 Skill 包（SKILL.md + 资源）取代。

当前结构化层**只产 manifest 软字段、没有真 Skill 包生成**。直接删 `skill_set` 会丢掉唯一的「技能」表达。故先建 Skill 包生成（写 SKILL.md + 落对象存储 + 落能力行的 `skill_*` 列），再把 `skill_set` 从软字段集与 `ManifestSchema` 移除。

## 二、结构坍缩计划（多步多表 → 2 步一结构）

这是「对齐」的主体，也是最大、最具破坏性的一块（删域、删表、churn 数百测试）。**含不可逆的删表，建议逐步、各步保持测试绿、按下面顺序评审推进，不一次性盲拆。**

现状：`import / extract / structure / publish / drafts` 五域，`raw_snapshots`、`session_segments`、`capability_candidates`、`candidate_evidence`、`capabilities`、`capability_versions`、`skill_packages`、`publications`、`capability_tiers`、`marketplace_listings`、`publish_batches`、`publish_batch_items`、`drafts` 共 13 表。

目标：`import` + `analyze-synthesize` 两域，`raw_snapshots`、`session_segments`、`capabilities`（合并）三表。

建议顺序（每步独立 PR、独立绿）：

1. 先落 Skill 包生成（解 P5 前置），新 `capabilities` 表设计冻结（含 `status: generating|draft|published|archived|failed`、`source_snapshot_id`、`manifest jsonb`、`manifest_hash`、`skill_*`、封面/可见性冻结列）。
2. extract + structure 合并为单个 `analyze-synthesize` job：一个任务里识别候选（内部态，不落候选表）→ 逐个合成能力行（status 从 generating 走到 draft）。候选/证据表退役。
3. 发布降为能力上的同步事务：`capabilities.status` draft→published + 冻结指纹/Skill 包 + 发 `capability.published`。删 `publish_batches/items`、`capability_tiers`、`marketplace_listings`、`publications`（市集/价格/批量/复审为非目标）。
4. 版本表 `capability_versions`、`skill_packages` 合并进 `capabilities`。
5. `drafts` 退役（2 步一次处理用不到；续传靠查最新快照 + 任务状态 + 能力）。
6. 导入完自动接力 `analyze-synthesize`（收尾事务建任务 + 巡查兜底）。
7. 同步 H1 / M2 真删 / P5 退役收尾。

每步都要同步改：对应 routes/handlers/repos、job 注册表、`db/migrations` 新增迁移、`__tests__` 对应用例、`@cb/shared` 契约。第 2~5 步删表不可逆，须先评审确认再做。

## 三、建议

- 本分支先合 M1（安全、已绿）。
- 结构坍缩按上面 1→7 顺序，每步独立 PR；删表步骤（2~5）动手前确认。
- H1 待前端排期一起做；M2 真删上线前做；P5 随第 1 步 Skill 包落地。
