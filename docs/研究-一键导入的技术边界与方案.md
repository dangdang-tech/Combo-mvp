# 研究 · 托管站「一键导入全部历史」的技术边界与方案

> 问题:用户打开公网站,**点一下就把本机 Claude/Codex 全部历史导上来,不手选文件夹**。技术上可解吗?

## 一、绕不过去的硬约束
**网页(浏览器 JS)无法静默读取磁盘任意文件。** 这是浏览器安全模型的根本边界——能背着用户读 `~/.claude` 的网页,也能读 SSH 私钥/钱包/密码。所以:
- ❌ 「打开网页自动抓取本机历史」= 不可能(也不该可能)
- ❌ 「mock 一个假授权然后读盘」= 同一个洞,堵死
- ❌ 服务器远程读用户机器 = 够不到

**能做的不是「免授权读盘」,而是「把授权压到最短」。**

## 二、三条真实路径对比
| | 静默读盘 | A. CLI 一行命令 | B. File System Access API |
|---|---|---|---|
| 导全部、不手选 | 不可能 | ✅ | ✅ |
| 用户动作 | — | 终端粘一行 | 首次点1次授权,复访0 |
| 平台/浏览器 | — | 全平台 | 仅 Chromium(Chrome/Edge/Arc/Brave) |
| 复用现有代码 | — | 高 | 高(复用 parseSession) |
| 体验「得劲」度 | — | 重(要开终端) | 轻(浏览器内,像点个授权) |

## 三、选定:B(FS Access)为主,A(CLI)为备
理由:用户要的「点一下→授权→完事」,**正是 FS Access 授权的形态本身**,不用造假授权。把它压到最短:
1. 点「一键导入」→ `showDirectoryPicker({startIn:"home"})` 弹框**已定位主目录** → 点「打开」(这一下=授权)
2. 拿到 home 句柄后**直接定位** `~/.claude/projects` 与 `~/.codex/sessions` 两个子目录读取
   —— **不遍历整个 home**(实测忽略 Downloads 等),只读这两处,快
3. `parseSession` 两格式自动识别 → 上传 → 自动进下一步
4. 句柄存 **IndexedDB**;复访 `queryPermission==='granted'` → **零点击自动导入**;要复确认 → 一下「允许」

**稳定态:首次两下(点按钮+点「打开」),以后零下。** 这是安全模型内的最短。

非 Chromium(Safari/Firefox):自动回退「选文件夹」(webkitdirectory);CLI 作为终极备用。
本地开发(localhost):仍用服务器直扫 `~/.claude`+`~/.codex`(无弹框,最省事)。

## 四、为什么不是「静默」也 OK
对开发者用户,一次「授权读取我的 AI 历史目录」的心智负担,等同于授权一个 OAuth——可理解、可信任、可撤销。比「开终端粘命令」轻,比「一个个选文件夹」短。

## 五、验证(本轮)
- 真实浏览器跑 `fsCollectHome`(mock 句柄):找到 claude+codex 各 1、codex project=cwd basename、**忽略 .claude/.codex 之外的文件**、进度回调正常 ✓
- 线上 Chrome:FS 检测 ✓、按钮文案 ✓、复访自动导入 no-op 无报错、**零 console 错误** ✓
- `parseSession` 两格式 + `/api/import-uploaded` 合并:前几轮已分别测过 ✓
- 唯一无法自动化:原生授权框的「打开」点击(用户手动那一下)
