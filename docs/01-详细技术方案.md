## 1. 架构总览

### 1.1 进程拓扑

「一库一镜像四入口」：同一仓库、同一镜像，按启动入口分四类进程。飞书 2.1 定义的是三入口（api / worker / consumer），其中 sweeper 是 worker 内「后台循环单活」的一部分；本期生产栈把 **sweeper 单独作为第四个入口进程**落地（与本地 v2 一致），因为它必须**全局单实例**，与可水平扩缩的 worker 生命周期解耦。逻辑上仍遵守飞书 2.1「worker = 长任务 + 全部后台循环（Redis 锁单活）」——只是把单活循环物理拆到 sweeper 进程，靠 Redis 锁保证单活。

| 进程入口     | 职责                                                                                                                                                                                                                                                                                                                              | 可扩缩                 | 关键约束                                                                                                                                                                                                      |
| ------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **api**      | 在线 REST + SSE；Builder 编排、发布门事务、聚合查询、消费读 API（预留）、SSE Hub（Redis Streams 桥接 + 补发 + 跟流）。**api 不跑任何后台循环**——这是「api 无状态、SSE 可重连任意实例」的前提                                                                                                                                      | 水平扩                 | 无进程内共享状态；幂等判定在 app 层                                                                                                                                                                           |
| **worker**   | 本期注册四类 Job processor：`import` / `extract` / `structure` / `publish_batch`（publish_batch 编排发布事务，并对一批 item 逐个 structure+publish、无连坐）；`runtime_gen` 与 `evaluate` **本期不交付**（schema 已冻结，processor 不注册）。经 LLM Gateway 出口调模型                                                            | 按 Job 类型扩缩        | 任务状态以 PG `jobs` 表为唯一真源（BullMQ 只触发）；**所有写入带 `WHERE job_id=? AND fence_token=?`（执行租约/fencing，见 1.3）**                                                                             |
| **consumer** | 本期两类域事件消费：`MarketplaceProjection`（capability.lifecycle → marketplace_listings）/ `NotifyConsumer`（notify.events → 站内 + 飞书/邮件）。`MeteringConsumer`（usage.metering → usage_events + daily 聚合）**本期不启动**（usage 置空）。每类单实例单线程按 outbox id 顺序拉取                                             | **不可多实例**（保序） | **运行时防重**：每个 topic consumer 启动拿 PG advisory lock（或 Redis lease），拿不到直接退出 / 只报 degraded、不消费；at-least-once + event_id 幂等；xid 提交序水位；cursor 更新与事件处理**同一事务内**完成 |
| **sweeper**  | 全局单活，**只做 job 对账 + orphan 清理 + outbox 兜底，不承担 consumer 保序**：① 扫 `jobs.lease_until` 过期的 running job 对账（lease 过期才递增 `attempt_no` 并以新 `fence_token` 重入队，旧 worker 写入因 fence 不匹配被拒）；② orphan 对象/产物清理；③ outbox 投递滞留巡查（告警 + 补投递）。session 热态归档随 Trial 上线再加 | **固定单实例**         | Redis 锁 + TTL 续期保证单活                                                                                                                                                                                   |

进程间不互调：api 与 worker/consumer/sweeper 只经 **PG**（任务/事件真源）、**Redis**（队列/控制/事件流）间接通信。P0 允许单机用 Docker Compose 把四进程合在一台机器跑（无进程内共享状态、配置外置），出现资源争抢再物理拆分。

### 1.2 数据流（导入 → 去敏 → 提取 → 结构化 → 发布）

端到端链路（对齐飞书 2.3 时序图 + PRD 第五章）：

1. **导入（STEP①，云端任务）**：前端申请 S3 预签名 URL → **原文分批直传 S3**（此阶段页面需在线、可断点续传）→ 传完 `POST /import/jobs`（引用已上传对象）→ api 建 `jobs(type=import)` + BullMQ 入队（jobId 幂等）→ 秒回 job_id。此后**可关页**。worker 从 S3 拉原文 → **解析 + 去敏 + 切段** → 先建去敏 `raw_snapshots`（新快照）、再写隶属该快照的 `session_segments`（**按 `(snapshot_id, content_hash)` 快照内去重**，原文处理完即弃、不落正式盘；重导生成新快照、旧快照及其段保留）→ 进度写 `jobs.progress` + XADD Redis 事件流 → 完成写 Outbox(notify)。**导入隐私口径（2026-06-15 拍板）**：全量上传原文 + 云端去敏；界面绝不出现「数据不出本机/仅上传精简数据」承诺（导入-05 负向 P0）。
2. **提取（STEP②，云端任务）**：`POST /snapshots/{id}/extract` → 萃取 Job 入队 → worker 经 LLM Gateway 聚类 → `capability_candidates` + `candidate_evidence` 落库 + XADD 候选流 → SSE 候选逐个浮现。单候选失败可 `POST /candidates/{id}/retry`，不阻塞其余。
3. **选择（STEP③，纯前端）**：从提取结果选定一个能力或「全部发布」。**不调模型、不写库**——选择结果只暂存在向导客户端态，随下一步请求带去。
4. **结构化（STEP④，云端任务）**：`POST /capabilities`（从选定候选建 draft 版本）→ `POST .../versions/{ver}/structure` → 结构化 Job → worker **直接读 `candidate_evidence / session_segments`**（本期结构化主链路不依赖 ExperiencePack）经 LLM Gateway 逐字段生成**软字段**（name/tagline/role/goal/instructions/skill_set/starter_prompts，一个存一个），**硬字段**（id/version/status=draft/inputs schema/output type/boundaries）平台锁定 → XADD 字段流（field_start/field_delta/field_done/field_stuck）→ SSE 字段流。卡住给三条退路，不裸转圈。
5. **发布（STEP⑤，事务，固定形态）**：`POST /versions/{ver}/publish`（幂等键）→ **单 PG 事务**：冻结 manifest 记 hash + `status=published`（即上架「Alpha 审核中」）+ 按发布定价固化 `capability_tiers` + 写 Outbox(`capability.published`) → MarketplaceProjection 拉取 → `marketplace_listings`（市集可见，标 alpha_pending）。**Alpha 人工评审**事务回写：通过→正常上架；拒绝→事务回退（有上一 published 版本则回退该版并发 `capability.published` 指回退版，无上一版则下架发 `capability.unpublished`），创作者侧显示**简单拒绝原因 + 重试/编辑入口**。批量发布走 `publish_batches`，每 item 独立幂等键/独立状态机，失败只标该 item、可单独重试。
6. **计量回流（本期置空，仅冻结 schema）**：runtime 生成在本期不交付，`usage.metering` 事件不产生、**MeteringConsumer 本期不启动**、daily 聚合**不跑**。`artifacts / usage_events / daily_*` 仅冻结表结构与事件 schema。**所有用量类指标（本月消耗、累计/调用量、活跃消费者、token 趋势、个人主页总调用量）本期统一显示「暂无数据 / 上线后填充」**，前端走占位态（非空错误、非裸转圈）。计量真源（event_id 幂等、effectively-once）的设计在表与 §9 差异中记录在案，留待上线时按已冻结 schema 接入。

### 1.3 并发三机制（飞书 4.6 / 6.1 / 本地 v2 2.7）

| 机制                                                                     | 解决什么                                                                                                                                     | 落地方式                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Tasks 去重 + 执行租约/fencing（任务真源 + jobId 幂等 + fence_token）** | 双击/断线重试/双标签页不产生重复任务；**sweeper 重入队后旧 worker 不会再写 completed/progress 覆盖新执行**（避免重复候选/重复字段/旧覆盖新） | PG `jobs` 表是任务状态唯一真源，BullMQ 只触发；写命令带 `Idempotency-Key`（判定在 app 层，落 `idempotency_keys` 表，含租约 `locked_at/expires_at`）；BullMQ `jobId` 去重。**执行租约/fencing**：`jobs` 增 `attempt_no / lease_owner / lease_until / fence_token`，worker 领取时写自己的 `lease_owner/fence_token` 并续 `lease_until`，**所有对 jobs 及其产物（candidates/字段/快照）的写入都带 `WHERE job_id=? AND fence_token=?`**；sweeper **仅在 `lease_until` 过期后**递增 `attempt_no`、换新 `fence_token` 重入队，旧 worker 因 fence 不匹配写入失败、安全退出；**取消走同一 fencing**（标 cancelled + 换 fence，确保被取消的旧执行无法再回写） |
| **Progress SSE（进度续传）**                                             | 永不裸转圈、断线接得上、超窗能恢复                                                                                                           | 事件源 = Redis Streams `events:{kind}:{id}`；`fetch-event-source` + `Last-Event-ID` 续传；超窗先推**按流类型定义的 `state_snapshot`**（job 流 = jobs.progress 全量；structure 流 = structure_state 全量；session 流 = 会话视图全量）再续流；api SSE Hub 补发 + 跟流                                                                                                                                                                                                                                                                                                                                                                                  |
| **Drafts 续传（断点续传）**                                              | 五步任一步中断可一键回到中断处、已生成内容不丢                                                                                               | 每步可存草稿；`capability_versions.structure_state`（每字段状态/已生成值/卡住时长）+ `jobs.progress` 持久化；工作台草稿条记录所处步骤与进度（如「结构化中 60%」），点回去落点对得上（外壳首页-33 P1）；已生成内容靠「字段级落库 + 中断保留已完成部分」保证不丢                                                                                                                                                                                                                                                                                                                                                                                       |

### 1.4 统一 AI 网关（LLM Gateway）

所有模型/embedding 调用仅经 LLM Gateway 出口（飞书 4.3 / 4.6）：**预算闸**（成本上限 + 匿名按 token 限流）、**超时分级**（40s/45s/60s/180s，MVP 实测值，按任务类型选档）、**重试 ≤ 2**、**审计落 `audit_llm_calls`**（成本审计，非计费真源）、**embedding 路由**、**SSRF 闸**（fetch_url 工具）。LLM Gateway 是 domain 层声明的两个端口之一（另一个是 ObjectStore），由 app 层注入 infra 实现 —— 这是六边形降级版（飞书 4.2）。

## 2. 后端模块拆分（目录划分建议）

四层目录约定 + eslint import 强制（飞书 4.1 / 4.2）。依赖只许向下：`api → app`；`app → domain`；`app → infra`（组合调用）；**`domain` 不 import 任何 infra**，需要外部能力时只依赖自己声明的端口接口（LLMGateway、ObjectStore），由 app 组装注入。

```
src/
  api/                      # 协议适配层：HTTP/SSE，只做 zod 校验 + 幂等键透传，无业务判定
    routes/
      import.routes.ts      # /import/jobs · /import/connect/pair · /upload
      jobs.routes.ts        # /jobs/{id}/events(SSE) · /cancel
      extract.routes.ts     # /snapshots/{id}/extract · /candidates/{id}/retry
      capabilities.routes.ts# /capabilities · /versions/{ver}/structure · /manifest
      publish.routes.ts     # /versions/{ver}/publish · /publish-batches（/trial-pass · /evaluate 仅 schema 预留，本期不挂可调用实现）
      dashboard.routes.ts   # /dashboard/* 聚合
      profile.routes.ts     # /creators/{id}/profile · /follows · /likes
      # runtime.routes.ts —— 本期不交付：runtime 表/事件协议仅冻结 schema（B-40），无可调用 /runtime API
      market.routes.ts      # /market/listings · /market/apps/{slug} · /market/manifests/{ver}（消费读 API 预留）
    sse/                    # SSE Hub：Redis Streams 桥接、补发、Last-Event-ID 续传、state_snapshot 分型
    middleware/             # 鉴权(JWT)、限流、错误 envelope（绝不裸露错误码）
  app/                      # 编排层：用例编排 + 事务边界 + 组合 infra + 注入端口
    builder/                # 五步用例：importService / extractService / structureService / publishService
    dashboard/              # 工作台聚合用例
    profile/                # 个人主页聚合用例
    jobs/                   # jobs 生命周期编排、取消语义、BullMQ↔jobs 同步协议
    idempotency/            # 幂等键判定（租约、request_hash、response_ref）
  domain/                   # 领域层：纯业务，零 infra 依赖
    capability/             # 能力体/版本状态机（draft→published→superseded/review_rejected）
    manifest/               # App Identity 软硬字段模型 + manifest_hash + 校验
    extract/                # 萃取 pipeline 核心（沿用 MVP v3 锚定 taxonomy 算法）
    redaction/              # 去敏规则引擎（新增能力，导入最需谨慎的一块）
    parsing/                # 会话解析（沿用 MVP 解析真源，唯一一份）
    publish/                # 发布门、tiers 固化、评审状态机
    ports/                  # 端口接口：LLMGateway、ObjectStore
  infra/                    # 基础设施实现
    pg/                     # PG 查询、迁移、事务封装、outbox 写入（与业务同事务）
    redis/                  # 两实例客户端：redis_queue(BullMQ) + redis_hot(Streams/控制流/锁/限流)，连接串/健康检查分开
    bullmq/                 # 队列定义、worker 注册、jobId 幂等
    s3/                     # MinIO 客户端、预签名 URL、orphan 清理（ObjectStore 端口实现）
    llm/                    # LLM Gateway 实现：预算闸/超时分级/重试/审计/embedding/SSRF（LLMGateway 端口实现）
    auth/                   # Logto 自托管：OIDC 回调换 JWT、issuer/JWKS 校验、单账号双角色、share_token 匿名
    outbox/                 # outbox 写入封装 + xid 水位（写在 pg/ 旁，consumer 复用读取）
  workers/                  # worker 进程入口：本期注册 import/extract/structure/publish_batch 四类 processor；runtime_gen/evaluate 仅 schema 预留、不注册
  consumers/                # consumer 进程入口：本期两类（MarketplaceProjection/NotifyConsumer）；MeteringConsumer 仅注册不启动（usage 置空）
  sweeper/                  # sweeper 进程入口：jobs 对账 + orphan 清理 + outbox 兜底
  shared/                   # zod schema(即 OpenAPI 真源)、错误码 envelope、SSE 事件类型、常量
  bin/                      # 四入口启动脚本：server.ts / worker.ts / consumer.ts / sweeper.ts
```

eslint 依赖规则（强制，违反即 CI 失败）：`domain` 不得 import `infra`/`app`/`api`；`app` 不得 import `api`；`api` 不得 import `infra`（只经 app）；worker/consumer/sweeper 不得 import `api`。

## 3. 前端模块拆分

React 18 + TypeScript + Vite；两个应用壳 `/creator/*`（创作者中心）与 `/a/:slug`（Runtime，消费端可独立加载）；状态 TanStack Query（服务端态）+ Zustand（向导跨步骤客户端态）；流式 `fetch-event-source`（SSE + Last-Event-ID）；图表 ECharts；契约 OpenAPI 3.1 codegen TS client；组件 shadcn/radix 基座 + Agora 主题层。

```
web/src/
  app-shell/                # 外壳 Shell（D14：五步流程中外壳恒定不变）
    Sidebar/                # 侧栏：创作组(工作台/我的能力/上传能力/数据分析/收益) + 我的组(个人主页)；收起/展开 + 偏好持久化
    Topbar/                 # 顶栏面包屑 + 账号头像
    layout/                 # 三段式布局壳；当前页选中高亮
  pages/
    dashboard/              # 工作台：经营摘要 + 4 指标卡(含活跃消费者，usage 类本期占位) + token 趋势(占位) + 能力表格(sparkline/收益占位) + 草稿条 + 时间范围切换
    profile/                # 个人主页（全六分区 P0）：身份区/指标带/能力密度榜/热力图/能力网络缩略(无展开)/作品墙(全只读，usage 类占位)

  builder/                  # 五步上传向导（共用外壳，只换中间内容区）
    WizardShell/            # 步骤条状态语义(已完成/进行中/待办/异常) + 底栏导航 + 保存草稿
```

### 6.2 迁移顺序（硬性）

先起依赖 → 迁数据库 → 迁移成功才起业务容器（避免新代码碰旧表）。迁移**只加不减、向后兼容**（保证回滚安全）。

**固定启动顺序（硬性）**：`postgres → logto seed/alteration → logto → 业务迁移 → 业务容器`。Logto OSS 不自跑迁移，必须先 CLI seed 再把 alteration 作为单实例 job 跑，跑完才起 logto 运行态。

```bash
# 1. CI 产出不可变镜像（按 git-sha 打标签，生产机不现场构建）
# 2. 先起基础依赖（PG + Redis 两实例 + MinIO）
docker compose up -d postgres redis_queue redis_hot minio
# 3. Logto 建表/迁移（一次性容器；postgres healthy 后跑，跑完退出；Logto OSS 不自跑迁移）
docker compose run --rm logto_db_seed      # CLI: logto db seed（首次部署建表 + 初始数据，幂等可重入）
docker compose run --rm logto_alteration   # CLI: logto db alteration deploy（单实例 job 跑 schema alteration）
# 4. 起 Logto 运行态（自身不建表，依赖上一步完成）
docker compose up -d logto
# 5. 业务迁移（失败即止，不起业务容器；与 Logto 迁移各自独立 schema/库、互不干扰）
docker compose run --rm migrate npm run migrate
# 6. 迁移成功后起业务容器
docker compose up -d --no-deps api worker consumer sweeper web
# 7. 健康检查（含 Logto OIDC discovery 可达且 issuer/jwks_uri 正确）
curl -f http://localhost/api/v1/ready
```

### 6.3 健康检查口径

- **真失败（影响 /ready，应用等就绪才启动、不就绪即不可用）**：PostgreSQL、Redis（**两实例 `redis_queue`/`redis_hot` 均需可达**）、MinIO、**Logto OIDC**（拉 `LOGTO_ISSUER` 的 discovery document `GET {LOGTO_ISSUER}/.well-known/openid-configuration`，断言返回里 `issuer` 与 `jwks_uri` 存在且 `issuer` 匹配；不用 `{LOGTO_ENDPOINT}/api/.well-known/...` 这套错误路径）任一不可达。
- **degraded（不算失败、不停服）**：外部 AI（OpenRouter）不稳定 → 只标记状态变差，对外服务继续；LLM Gateway 对调用方返回降级（进度短语 + 退路），不裸转圈、不裸错误码。这与飞书 4.1「可观测性」+ 验收三条硬规则一致。

### 6.4 启动脚本与集成脚本规划（位于仓库根 `scripts/` / `infra/`）

- `scripts/dev-up.sh`：本地起依赖 + 四进程（合跑或分跑）。
- `scripts/migrate.sh`：迁移封装（先依赖后迁移）。
- `scripts/smoke.mjs`：端到端冒烟（建账号→导入→萃取→建能力→结构化→发布→读工作台/主页）。
- `scripts/integration/*`：分模块集成脚本（与 O-05 对应）。
- `infra/docker-compose.yml` + `infra/Dockerfile.api` + `infra/Dockerfile.web`（O-01/O-02）。

3. **计量回流 / usage —— 本期置空**。`usage_events / daily_*` 仅冻结 schema，**MeteringConsumer 不启动、daily 不聚合、不产生任何用量数据**（B-36）。所有用量类指标（本月消耗、累计/本月调用、活跃消费者、token 趋势、个人主页总调用量、作品墙调用次数、收益/分析金额）统一显示「**暂无数据 / 上线后填充**」占位态（非空错误、非裸转圈，B-32/F-05/B-33/F-06/F-07）。「未入账事件 N 条」机制随计量上线再启用。
4. **个人主页 —— 全六分区 P0**。B-33/F-06 由 P1 升为本期必做 P0，六分区（身份/指标带/能力密度榜/热力图/能力网络缩略/作品墙）逐项落表/API/UI；能力网络仅缩略、无展开入口；**P0 能力网络缩略不依赖 embedding**，用 session/tag 共现即时生成缩略关系（B-33），**B-37（embedding+capability_relations）保持 P1、限定为后续搜索/推荐增强，不作为本期主页缩略数据源**（消除依赖冲突，§9 第 9 条同口径）；usage 类指标按第 3 条占位。
5. **批量发布 —— 无连坐 P0**。B-29 升 P0：每 item 独立 idempotency key + 独立状态机 + 失败只标该 item（不连累其余）+ 可单独重试该 item；前端 F-14 给批量结果列表与失败项「去补齐」回向导入口。依赖补齐 B-09/B-25/B-26/B-27/B-28。
6. **Auth —— Logto 自托管唯一**。不再 Logto/Supabase 二选一；compose 起 `logto` 容器，issuer/JWKS/回调域名/env/初始化/健康检查见 B-08 与 §6.1/6.2/6.3。

> 上述六项与文首「六项已拍板决策」完全一致，全文据此自洽。本节不再保留任何「需拍板/需确认」项。

**技术风险（非阻塞，按风险管理推进）**：去敏规则（B-17）是新增能力、无 MVP 资产、且是隐私安全关键路径，需单独设计规则集 + 高覆盖单测（导入隐私口径已统一为「全量上传 + 云端去敏」，去敏正确性是唯一隐私防线）；MVP 算法平移（解析 B-18、萃取 B-22、结构化软字段生成 B-25）需保持算法资产不退化（萃取 Jaccard 0.852 基线）；批量发布对一批逐个跑结构化耗时较长，靠 SSE 进度 + 每 item 独立状态规避「裸转圈」与「一败全败」。

## 9. 与本地 技术方案-v2 的差异（逐条，以飞书为准）

| #   | 维度                      | 本地 v2                                                       | 飞书（权威）                                                                                                                                    | 本方案采纳                                                                                                                                                                                                                                                                                                                                    |
| --- | ------------------------- | ------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | 进程数                    | api/worker/sweeper/consumer **四进程**（sweeper 独立列）      | api/worker/consumer **三入口**，sweeper 是 worker 内「后台循环单活」                                                                            | **取飞书逻辑 + v2 物理拆分**：sweeper 物理独立第四进程，但仍守「单活循环」语义（Redis 锁），因 sweeper 必须全局单例、与可扩缩 worker 解耦。已在 1.1 注明。                                                                                                                                                                                    |
| 2   | Job 类型数                | worker 跑「**四类**」任务（导入/提取/结构化/发布）            | **五类**（import/extract/evaluate/publish_batch/runtime_gen）                                                                                   | 取飞书类型定义；**本期实际注册四类 processor**：import/extract/structure/publish_batch；**runtime_gen（试用不做）与 evaluate（Auto-Eval 不进本期路径）仅冻结 schema、不注册 processor**。                                                                                                                                                     |
| 3   | 事件总线提交序安全        | 仅「按没处理过的先处理 + 事件去重」，**未提 xid 提交序水位**  | 明确 **xid8 提交序水位**：consumer 遇首条 xid≥xmin 即停、不跳 unsafe 行、cursor 永不跨 in-flight 事务                                           | 取飞书 xid 水位（v2 的朴素顺序消费在 BIGSERIAL 分配序≠提交序时会漏读 in-flight）。B-13/B-14 落地。                                                                                                                                                                                                                                            |
| 4   | 毒丸/失败策略             | 笼统「事件去重不漏」                                          | **按 topic 分策略**：lifecycle 卡住等人工；metering/notify 进 dead_events + cursor 跳过 + 收益页/成本日报显式暴露「未入账 N 条」                | 取飞书分 topic 策略（B-15）。                                                                                                                                                                                                                                                                                                                 |
| 5   | manifest 结构             | 「沿用现有**扁平结构 manifest v0.1**，软/硬只是界面分组标记」 | **App Identity 软硬分层**：软字段经验体生成、硬字段平台锁定，status=draft 是硬字段                                                              | 取飞书软硬分层为 manifest 一等结构（不只是 UI 标记），但软字段值仍可映射到 MVP 蒸馏字段。B-25/B-26 按软硬分层落地。                                                                                                                                                                                                                           |

| 8   | 经验体 ExperiencePack     | v2 **未提**                                                   | 飞书：冻结 schema、从 candidate_evidence 蒸馏 taste/rules/cases；右栏三面板                                                                     | 补入（B-38）：**本期仅冻结 schema、不参与结构化主链路**；为避免依赖倒挂，B-25 改为直读 candidate_evidence/session_segments，不依赖 B-38（见第 8 节已收口第 12 条）。                                                                                                                                                                          |
| 9   | 能力关系图谱/embedding    | v2 称「向量检索属消费端，本期不做」                           | embedding 写路径**直接触发异步任务（不走事件总线）**，支撑能力关系边 + 市集搜索                                                                 | **拆两层**：① 本期个人主页能力网络**缩略**不依赖 embedding，用 session/tag 共现即时生成缩略关系（B-33，P0，不读 capability_relations）；② embedding + capability_relations（B-37）保持 P1，**限定为后续搜索/推荐增强**，不作为本期主页缩略数据源。如此既满足主页 P0 缩略，又不把 B-37 拽进本期，消除「P0 主页缩略无可交付数据源」的依赖冲突。 |
