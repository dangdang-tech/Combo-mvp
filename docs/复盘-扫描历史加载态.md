# 复盘 · 扫描历史的分阶段加载态

> 流程:设计 → 开发 → 测试 → 复盘。本次「先设计后开发」(用户明确要求)。

## 设计(先)
扫描旧态只有一行干文字(`本地解析 N 个会话…`)+ 按钮转圈,不够友好。出了三个方向给用户选:
- **A. 分阶段清单**(选中):授权→扫描→解析→上传逐个点亮 + 来源 chip + 进度条 + 隐私行。状态最清楚、能安抚焦虑、未知总数时也优雅。
- B. 实时文件流(终端感);C. 动效计数+骨架。
依据:导入是**多阶段**过程,用户最大的焦虑是「它到底在干嘛/会不会偷我文件」。A 把阶段显性化 + 隐私行直接回应,信息密度恰当。

## 开发
- `importLoader(phaseDefs)` 控制器:`enter`(点亮当前、前序打勾)/`sources`/`progress`/`note`/`indet`/`fail`(带「重新开始」)。纯 DOM,贴现有 token(`--ember`/`--ink*`/`--line`、来源色 claude#C96442·codex#10a37f、新增 `spin`/`slide` 动画)。
- `fsCollectHome` 改为**先枚举计数(不读内容,快)再读取** → 真实进度条;回调 `{done,total,parsed,sources}`。
- 三条导入路径(FS 授权 / 选文件夹 / 本地直扫)统一驱动加载态。
- 选文件夹改用**动态 input**(不依赖常驻 DOM)—— 顺手修了潜在 bug:`showImported` 替换 `#imp` 后原 `#dir` 被销毁,旧「➕ 再导入」点了没反应。

## 测试(真实浏览器,chrome-devtools)
- 阶段机:auth→parse→upload 图标/类正确流转 ✓
- 进度:mock 句柄跑通,bar 3/3=100%、chip Claude 2 + Codex 1 点亮、计数文案对 ✓
- 失败态:err 类 + 标题 + 消息 + 「重新开始」按钮已接 `step1`,点击恢复导入按钮 ✓
- 截图:与设计稿 A 一致 ✓
- 线上(railway):加载态渲染、4 阶段、bar 70%、计数对、**零 console 错误** ✓

## 还可做
- 大文件夹(几百 MB)解析时主线程可能卡 → Web Worker 分批(后续)。
- 阶段微文案可随真实耗时轮换(如「正在读取第 N 个项目…」)。

## 一句话
把「一行干文字」升级成「会讲故事的分阶段清单」:每一步在干嘛、找到了什么、进度多少、隐私如何,一眼可见;失败有退路。先设计定方向、后开发落地、真实浏览器验收。
